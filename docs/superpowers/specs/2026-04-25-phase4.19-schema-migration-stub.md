# Phase 4.19: Schema Migration / Promotion System (stub)

**Date:** 2026-04-25
**Status:** **Superseded** by `2026-04-26-phase4.19-event-promoter-design.md`. The full design diverged from this stub's chain-of-single-step-promoters framing — the lib now treats the promoter as an opaque caller-supplied callback (no registry, no composition, no direction enforcement) per user direction during the 2026-04-26 brainstorm. Kept for historical context only; do not implement against this file.
**Depends on:** Phase 4.16 (`entry_type_version` recorded on every event).
**Requirements:** see the design spec.

## Intent

Once events carry an authoring schema version, the lib needs a system that can read older events through a chain of promoter functions and present them in the current schema's shape — without rewriting the event log (the log is append-only and tamper-evident).

## Anchor points (already established, do not relitigate in 4.19's design)

- Project memory `project_event_schema_migration_strategy.md`: chain-of-promoters pattern, forward + rollback, read-time promotion only, event log untouched.
- `transformVersion` on `WirePayload` already records the wire transform's version per FIFO row (REQ-d00122-D); 4.19 is a different axis (authoring schema, not wire shape).
- Existing read-side surfaces that will need to consult the promoter chain:
  - `Materializer.fold` (per-aggregate replay during materialized-view rebuild)
  - `EventStore.append`'s upcasting path (if a stale promoter-aware caller receives an old-version event from `findAllEvents` and re-emits)
  - Any future portal-side query API that reads historical events

## Scope sketch (to be properly designed in 4.19's brainstorm)

- Promoter registry: `(entry_type, fromVersion, toVersion) → (Map<String,Object?>) → Map<String,Object?>` plus an inverse for rollback.
- Resolution: at read time, look up the chain `authoringVersion → currentVersion`, apply in order. Cache resolved chains per entry-type per process.
- Failure modes: missing promoter (schema gap in code), promoter throws, chain doesn't reach current version. All should surface as a typed exception, not silent data loss.
- Materializer interaction: rebuild walks events in sequence order; each event runs through promoters before being handed to the materializer's fold. Old materialized-view rows may need rebuild after a promoter chain extends.
- Test surface: round-trip (forward then rollback), missing-promoter detection, multi-hop chains, performance on rebuild over many events.

## Why this is a separate phase from 4.16

4.16 only adds the `entry_type_version` field and writes it on append. That is required infrastructure but not the migration system itself. Building the promoter chain on top of 4.16 is a meaningful surface (registry shape, error model, materializer integration, test matrix) and deserves its own brainstorm + spec + plan rather than being smuggled into 4.16.

## Non-goals

- 4.19 does NOT alter the event log. Promoters run at read time only.
- 4.19 does NOT introduce a "data migration" tool that rewrites events.
- 4.19 does NOT cover schema *additions* that don't change existing event shapes (those are no-op promoter chains).
