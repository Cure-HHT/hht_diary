## 1. Problem (unchanged summary from Phase 4.11 §1.4–1.6)

The `event_sourcing_datastore` library exposes `Future<List<…>>` reads only — no reactive / streaming / changefeed primitive. The example app's three read-side panels compensate with independent 500 ms `Timer.periodic` polls, each doing full scans on every tick:

- **`detail_panel.dart:59, :124`** — `findAllEvents(limit: 100000)` per tick. The `100000` is a "please don't page me" cap, not a real bound. At scale, re-materializes the whole event log every 500 ms.
- **`event_stream_panel.dart:31`** — `findAllEvents(limit: 500)` per tick, then sorts descending. Fine at demo scale, unbounded cost at production scale.
- **`fifo_panel.dart`** — after Phase 4.11 this panel is migrated to `listFifoEntries` + `findEventById`, but still polls on a timer because nothing tells it "the queue changed."
- N open panels × 2 Hz × independent `scheduleOf` + `findAll…` reads. No shared poller, no shared cache, no coordination point across consumers.

At mobile scale (hundreds of events, small number of panels) this is fine. At a hypothetical ops-dashboard scale (multi-day event logs, many panels, multiple destinations) it's quadratic in panel count and linear in log size per tick.

## 2. Why this is a stub, not a Phase 4.11 addition

Reactive/streaming requires design decisions that Phase 4.11 deliberately skipped:

1. **Concurrency contract** — Dart `Stream`? `ValueListenable`? `ChangeNotifier`? All three have different subscription and teardown semantics. Library hasn't chosen.
2. **Change-detection mechanism** — sembast has store-change notifications (`StoreRef.query().onSnapshots`); relying on them couples the library's public contract to sembast. An in-memory broadcast layer over the library's own write paths avoids the coupling but doubles the work and risks missing writes that happen outside the library (tests, ops scripts, sync-through ingest).
3. **Backpressure / delivery guarantees** — does a stream replay missed events for a late subscriber? Guarantee ordering vs `findAllEvents` results? What happens when the log is compacted (not planned, but the contract has to say something)?
4. **Scope of reactivity** — which reads are observable? Event log? FIFO state per destination? Materialized view? All of them? Just the ones consumers have asked for?
5. **Coordination primitive (deficit 1.6)** — separate question: does the library provide a shared `DataStore` handle that coordinates reads across consumers, or is coordination entirely the caller's problem? The reactive API shape informs this.

None of these decisions has been made. Making them inside the 4.11 PR would blow up its blast radius and force reviewers to approve a concurrency contract alongside two additive read methods.

## 3. Inputs for the eventual brainstorm

When this stub is taken up, the brainstorm should cover at minimum:

- **Landscape review**: how `sembast`, `isar`, `drift`, and comparable Dart persistence libraries expose reactive reads. What the `riverpod` / `provider` / `bloc` ecosystem expects from a data layer. What SwiftData / Room expose on their respective platforms as comparison points.
- **Use-case enumeration**: which specific UI patterns need to be reactive today (mobile app read screens, example-app panels, any portal work Phase 5+ introduces)? Which can stay polling? The deficit is real only where the UI actually needs sub-second freshness.
- **Scope decision**: per-aggregate watch? Global event-log stream? Per-destination FIFO stream? Named-query snapshots? Cheapest coverage of the real use cases, not the full Cartesian product.
- **Sync-through implication**: Phase 4.9's `EventStore.ingest` is a second write path. Any change-notification mechanism has to fire on both `append` and `ingest` or it'll drop events that arrived via sync.
- **Test harness** — reactive APIs are harder to test than `Future` APIs. What does "the stream emitted the expected sequence" look like as a test?

## 4. In-scope sketch (subject to the brainstorm overriding)

Rough shape if the brainstorm lands on a minimal reactive surface:

- `StorageBackend.watchEvents({int? afterSequence}) → Stream<StoredEvent>` — emits each new event as it's appended or ingested, starting after `afterSequence` (or from current tail if null). Closes on backend disposal.
- `StorageBackend.watchFifo(String destinationId) → Stream<List<FifoEntry>>` — emits the current queue snapshot on every change to any row in the destination's FIFO. Debounced at some frequency TBD.
- `StorageBackend.watchEntry(String aggregateId) → Stream<DiaryEntry?>` — emits the current materialized row for one aggregate whenever the materializer writes it.

All three are narrowly-scoped observations, not a general-purpose "watch any query." A general-purpose watch is the larger design.

## 5. Out-of-scope even for Phase 4.12

- Offline-friendly replay / event-log compaction.
- Cross-device subscription (clients watching a remote server's log). Orthogonal to sync-through ingest.
- Live queries over the materialized view beyond per-aggregate. Probably folds into the same design but isn't required to close deficits 1.4–1.5.
