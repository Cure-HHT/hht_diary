# Outgoing Native-Format `wire_payload` Optimization (Phase 4.13 Stub)

**Date**: 2026-04-24
**Ticket**: CUR-1154 (continuation — may split into its own ticket when fleshed out)
**Phase**: 4.13 — library-only; symmetric dual of Phase 4.9's ingest-side design.
**Status**: **STUB** — surfaced during the Phase 4.9 brainstorm (2026-04-24), carved out to keep 4.9 reviewable. Not ready to execute.
**Parent spec**: `2026-04-24-phase4.9-sync-through-ingest-design.md` (defines the `esd/batch@1` canonical batch format on which this optimization hinges).

---

## 1. Problem

REQ-d00119-B today mandates that every FIFO entry carry a non-null `wire_payload` — the serialized bytes that will be sent to the destination. The field exists because:

- **Retry determinism**: a sent-but-unacked batch is re-sent verbatim, not re-serialized from events (different serialization procedures or canonicalization bugs could produce different bytes on retry, breaking idempotency on the destination).
- **3rd-party wire formats are lossy**: sponsor-specific CSV, Rave EDC XML, and similar formats are NOT round-trippable from stored events — once you've serialized events into a sponsor CSV, you cannot losslessly reconstruct the events. The bytes must be kept for the send-log.
- **Send-log integrity**: the FIFO row IS the record of "we sent these bytes at T." `wire_payload` is that record.

Phase 4.9 defines a canonical batch format — `esd/batch@1` — that the library owns and that IS round-trippable: given the events and the envelope metadata, the canonical bytes are deterministically reconstructible (JCS canonicalization guarantees byte-equality across serialization runs on the same inputs). For FIFO entries whose `wire_format == "esd/batch@1"`, storing `wire_payload` duplicates data that is losslessly recoverable by re-serializing from `event_ids`.

On mobile, the highest-volume destination is the diary server, which is expected to speak the library's native format. Every FIFO row bound for the diary server currently stores `wire_payload` that is redundant with `event_ids` + the native envelope. At steady-state ingest volume this roughly doubles FIFO-entry storage for the primary destination.

At portal scale (if/when the portal runs outbound FIFOs of its own — e.g., portal-to-EDC relays), the same optimization compounds.

## 2. Why this is a stub, not a Phase 4.9 extension

The optimization is genuinely additive and pairs cleanly with 4.9's batch format, but including it expands 4.9's review surface meaningfully:

1. **REQ-d00119-B amendment** — relaxing `wire_payload` from required to conditional-on-wire-format is a spec change on an existing, shipped REQ (not a new REQ). Triggers coordination with any tests that assumed `wire_payload` is always present.
2. **`drain` behavior change** — the per-FIFO-row send path today reads `wire_payload` and ships. It would need to branch: native rows serialize-on-demand from `event_ids`; 3rd-party rows use `wire_payload`. Touches the retry path, which is load-bearing.
3. **Retry determinism** — the retry story is "re-send the same bytes." With serialize-on-demand, retries re-serialize fresh. This is safe for the canonical format (deterministic by construction) but the argument needs to be spelled out and tested.
4. **Migration of existing FIFO rows** — greenfield assumption means there's nothing in the wild, but any pre-existing FIFO rows on a dev machine would carry `wire_payload` that the new code path may or may not read. Backwards-compat shim is not needed at the schema level (null is a valid transition value), but the behavior needs to be correct for both shapes during the transition tick.

Scope-wise: Phase 4.9 is already substantial (ingest contract + two chains + canonical batch format + verification + 4 new ProvenanceEntry fields + 2 new REQs + 13 new test files). Adding REQ-d00119 amendment + drain refactor + tests is another half-phase of work. Reviewing them together makes each harder to reason about.

## 3. Design sketch (subject to brainstorm)

Rough shape if the brainstorm lands here:

**REQ-d00119-B amendment**: `wire_payload` is `String?` (nullable). Non-null required when `wire_format != "esd/batch@1"` (3rd-party lossy formats); MAY be null when `wire_format == "esd/batch@1"` (native format — reconstructible from `event_ids`).

**`drain` branch**:
```
if fifoEntry.wire_format == "esd/batch@1" and fifoEntry.wire_payload == null:
    # Serialize on demand from event_ids.
    events = [backend.findEventById(id) for id in fifoEntry.event_ids]
    bytes = esd_batch_v1_serialize(events, envelope_metadata)
else:
    bytes = fifoEntry.wire_payload   # 3rd-party or legacy row
send(bytes)
```

**Envelope metadata for on-demand serialization**: the batch envelope (`batch_id`, `sender_hop`, `sender_identifier`, `sender_software_version`, `sent_at`) must be available at drain time. Options: (i) store envelope metadata in the FIFO row (modest extra storage, still much smaller than `wire_payload`); (ii) reconstruct from the FIFO row's own metadata + `EventStore.source`; (iii) generate fresh envelope on each retry (breaks determinism — same `batch_id` must be reused across retries or the destination sees a new batch).

Option (i) is probably cleanest: small per-row overhead, deterministic across retries. Final call during the real brainstorm.

**Size estimate**: for a native `esd/batch@1` row with 10 events of average size 500 bytes, `wire_payload` is ~5KB; envelope metadata is ~200 bytes. Net saving ~4.8KB per row. At 100 rows/day on a mobile diary, ~500KB/day; compounds over the retention window.

**Depends on Phase 4.11**: serialize-on-demand needs `findEventById` (Phase 4.11 adds this). Without it, the lookup is O(N) per FIFO row at drain time. Phase 4.13 can precede Phase 4.11 only if we add a temporary in-memory cache of events by ID; cleaner to order Phase 4.13 after Phase 4.11.

**Depends on Phase 4.9**: the canonical batch format itself. Phase 4.13 cannot happen before Phase 4.9.

## 4. Scope for Phase 4.13

In scope:
- REQ-d00119-B amendment: `wire_payload` conditional on `wire_format`.
- `drain` branch: serialize-on-demand for native format; use `wire_payload` verbatim for 3rd-party.
- FIFO-row shape: optionally store envelope metadata per row (decide during brainstorm).
- Tests: round-trip determinism (serialize twice, bytes equal); drain behavior for native and 3rd-party rows; retry determinism.
- No new REQs claimed (extension of REQ-d00119; possibly an assertion added).

Out of scope:
- Portal-side outbound FIFO (portal doesn't originate; this is a mobile-only optimization in Phase 4.13).
- Dynamic wire-format negotiation with destinations (a destination announces it speaks native or 3rd-party at registration time).
- Compression of native `esd/batch@1` payloads (orthogonal optimization; if adopted, applies to both storage and wire).
