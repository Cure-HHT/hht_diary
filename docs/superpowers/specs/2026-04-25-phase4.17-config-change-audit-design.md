# Phase 4.17: Config-Change Audit Events

**Date:** 2026-04-25
**Scope:** `apps/common-dart/event_sourcing_datastore/` (lib + tests).
**Related:** Builds on REQ-d00138's existing reserved-system-entry-types pattern (`security_context_redacted/compacted/purged`). Reuses Phase 4.16's `entry_type_version`/`lib_format_version` machinery (lib stamps version 1 for system entry types).

## Summary

Every runtime mutation to lib-controlled configuration (destination registration, schedule changes, deletion, wedge recovery, retention policy application) emits a system audit event into the same `event_log` as application data. Each emission is synchronous in the same backend transaction as the mutation it records — atomic. Per-resource aggregate IDs make "show me everything that's happened to Primary" queryable as a single aggregate stream. The implementation reuses the existing `kReservedSystemEntryTypeIds` pattern; system entry types' `entry_type_version` is lib-stamped per Phase 4.16.

## Architecture

```text
caller -> registry.<mutation>(id, ..., initiator: i)
  |
  v
backend.transaction((txn) async {
  do the mutation                    <-- existing code path
  eventStore.appendInTxn(             <-- new: emit system event
    txn,
    entryType: 'system.<name>',
    entryTypeVersion: <lib-fixed>,
    aggregateId: '<resource>:<id>',
    eventType: 'finalized',
    data: { ...mutation-specific... },
    initiator: i,
  );
});

  Either both writes commit together, or neither does.
  System event participates in the same hash chain as application
  data, RLS, audit panel, BatchEnvelope wire propagation.
```

**Invariants:**

- All seven mutation surfaces emit system events synchronously, in the same backend transaction, or rollback together.
- Caller-supplied `Initiator` is required for operator-driven mutations (`addDestination`, `setStartDate`, `setEndDate`, `deactivate`, `delete`, `tombstoneAndRefill`).
- Lib-internal mutations (retention sweeps) use `AutomationInitiator('retention-policy-sweep')` automatically — no caller-supplied initiator on those.
- Per-resource aggregate IDs follow REQ-d00145-I's `<role>:<id>` precedent: `destination:<id>` for destination lifecycle and wedge recovery; `security-retention` for retention sweeps.
- System entry types are not materialized to `diary_entries` (their `EntryTypeDefinition.materialize` is `false` per REQ-d00140-C, matching the existing `security_context_*` system events).
- `tombstoneAndRefill` moves from a top-level function to a method on `DestinationRegistry`. Greenfield = breaking-rename is fine.

## Components

### 1. New reserved system entry types

In `lib/src/security/system_entry_types.dart` (extend the existing file that defines `kSecurityContextRedactedEntryType` etc.), add seven new entry types:

| ID constant | Wire string |
| --- | --- |
| `kDestinationRegisteredEntryType` | `system.destination_registered` |
| `kDestinationStartDateSetEntryType` | `system.destination_start_date_set` |
| `kDestinationEndDateSetEntryType` | `system.destination_end_date_set` |
| `kDestinationDeletedEntryType` | `system.destination_deleted` |
| `kDestinationWedgeRecoveredEntryType` | `system.destination_wedge_recovered` |
| `kRetentionPolicyAppliedEntryType` | `system.retention_policy_applied` |

(Six new — five for destination lifecycle/recovery, one for retention. The deactivate path reuses `system.destination_end_date_set` since `deactivateDestination` is just `setEndDate(now)`.)

All seven are added to `kReservedSystemEntryTypeIds`. All seven get `EntryTypeDefinition` entries in the lib's internal system-types registration (during `bootstrapAppendOnlyDatastore`) with `materialize: false`, `widgetId: ''`, `widgetConfig: {}`, `registeredVersion: 1`. The lib stamps these definitions itself — they are not user-supplied, and they cannot be overridden by callers.

### 2. `DestinationRegistry` constructor change

`DestinationRegistry`'s constructor today takes only the backend. Add two required dependencies for emission:

```dart
DestinationRegistry({
  required StorageBackend backend,
  required EventStore eventStore,
  required Source source,
});
```

`bootstrapAppendOnlyDatastore` already constructs both before the registry, so threading them in is a one-line change at the bootstrap site.

### 3. Required `Initiator` on every mutating registry method

Each of these methods gains a **required** named parameter `Initiator initiator`:

- `addDestination(d, {required Initiator initiator})`
- `setStartDate(id, dt, {required Initiator initiator})`
- `setEndDate(id, dt, {required Initiator initiator})`
- `deactivateDestination(id, {required Initiator initiator})` (forwards to `setEndDate`)
- `deleteDestination(id, {required Initiator initiator})`
- `tombstoneAndRefill(id, fifoRowId, {required Initiator initiator})` *(new method, see §4)*

Compile-time enforcement, same pattern as Phase 4.16's `entryTypeVersion`. No silent default to `AnonymousInitiator`.

### 4. `tombstoneAndRefill` relocation: top-level function → registry method

Today `tombstoneAndRefill` lives in `lib/src/ops/tombstone_and_refill.dart` and is exported as a top-level function. Phase 4.17 moves the implementation onto `DestinationRegistry`:

```dart
class DestinationRegistry {
  // ...
  Future<TombstoneAndRefillResult> tombstoneAndRefill(
    String destinationId,
    String fifoRowId, {
    required Initiator initiator,
  }) async { /* ... */ }
}
```

The original top-level export goes away. The internal logic (in `tombstone_and_refill.dart`) can stay as a private helper called from the registry method, OR be inlined into the registry — either is fine; the registry method is the public surface either way. The existing REQ-d00144 traceability comments move with the implementation.

`event_sourcing_datastore.dart` exports drop `tombstoneAndRefill` from the top-level `show` list. Callers that imported `tombstoneAndRefill` directly from the lib will fail to compile until they switch to `registry.tombstoneAndRefill(...)`. Greenfield → fine.

### 5. Synchronous emission inside each mutation's existing transaction

Each registry method already opens (or runs inside) a `backend.transaction` for its mutation. The audit emission goes inside that same `txn` block, after the mutation's side effects but before the txn commits.

Pattern (illustrative, exact field set per `data` table below):

```dart
Future<void> setStartDate(String id, DateTime startDate, {required Initiator initiator}) async {
  return _backend.transaction((txn) async {
    await _doSetStartDateInTxn(txn, id, startDate);   // existing code, refactored for txn arg
    await _eventStore.appendInTxn(
      txn,
      entryType: kDestinationStartDateSetEntryType,
      entryTypeVersion: 1,
      aggregateType: 'system_destination',
      aggregateId: 'destination:$id',
      eventType: 'finalized',
      data: <String, Object?>{
        'id': id,
        'start_date': startDate.toUtc().toIso8601String(),
      },
      initiator: initiator,
    );
  });
}
```

This requires `EventStore.append` to expose a transactional companion `appendInTxn(txn, ...)` that performs the same work but participates in a caller-supplied transaction. (Today `EventStore.append` opens its own; Phase 4.17 splits the body into a public `append` that opens the txn and a public `appendInTxn` that doesn't.)

Per-emission `data` field set by entry type:

| Entry type | `data` fields |
| --- | --- |
| `system.destination_registered` | `id`, `wire_format`, `allow_hard_delete`, `serializes_natively`, `filter_entry_types[]` (or `null`), `filter_event_types[]` (or `null`), `filter_predicate_description` (or `null` — caller-supplied free-text describing the predicate, since lambdas can't be serialized) |
| `system.destination_start_date_set` | `id`, `start_date` (UTC ISO 8601) |
| `system.destination_end_date_set` | `id`, `end_date` (UTC ISO 8601), `prior_end_date` (UTC ISO 8601 or `null`), `result` (`closed`/`scheduled`/`applied`) |
| `system.destination_deleted` | `id`, `allow_hard_delete` (always `true` since the method throws otherwise) |
| `system.destination_wedge_recovered` | `id`, `target_row_id`, `target_event_id_range_first_seq`, `target_event_id_range_last_seq`, `deleted_trail_count`, `rewound_to` |
| `system.retention_policy_applied` | `policy_full_retention_seconds`, `policy_truncated_retention_seconds`, `events_truncated`, `events_purged`, `cutoff_full`, `cutoff_purge` |

### 6. Retention policy emission integration

Today `EventStore.applyRetentionPolicy` (per REQ-d00138) emits `security_context_compacted` and `security_context_purged` events on non-empty sweeps. Phase 4.17 adds a single per-sweep `system.retention_policy_applied` event that records the sweep itself (its policy values, what it changed, what it deleted) regardless of whether anything was changed (zero-event sweeps still get an audit row).

The existing `_compacted` / `_purged` events are unchanged — they record per-sweep totals as part of REQ-d00138's existing contract. The new `_applied` event records the policy *itself* and is emitted with `initiator: AutomationInitiator('retention-policy-sweep')`.

`AutomationInitiator` already exists per REQ-d00135 — it identifies non-human actors. Lib code uses it for sweeps; callers don't see it.

### 7. `bootstrapAppendOnlyDatastore` wiring

The bootstrap function constructs:

1. Backend
2. EntryTypeRegistry (now extended with the seven new system entry types added by the lib itself before user-supplied entry types)
3. EventStore (uses backend + entry-type registry + source)
4. DestinationRegistry (now also takes eventStore + source)
5. SecurityContextStore
6. SecurityRetentionScheduler (if applicable — lib internal)

Phase 4.17 changes (4): the registry constructor now takes `eventStore` and `source`. The bootstrap wires these in.

## Data Flow

Three flow patterns, all sharing the same atomicity guarantee.

### A. Operator-driven destination mutation

```text
caller -> registry.setEndDate(id, dt, initiator: UserInitiator('clinician-7'))
  |
  v
backend.transaction((txn) async {
  // mutation
  await _writePersistentSchedule(txn, id, endDate: dt);
  // audit emission, in same txn
  await _eventStore.appendInTxn(
    txn,
    entryType: kDestinationEndDateSetEntryType,
    entryTypeVersion: 1,
    aggregateType: 'system_destination',
    aggregateId: 'destination:$id',
    eventType: 'finalized',
    data: {
      'id': id,
      'end_date': dt.toUtc().toIso8601String(),
      'prior_end_date': priorEndDate?.toUtc().toIso8601String(),
      'result': result.name,
    },
    initiator: UserInitiator('clinician-7'),
  );
})
```

If the audit emission throws (e.g. registry not found, entryType not registered — both shouldn't happen in bootstrapped state), the entire transaction rolls back. The schedule mutation is undone with the audit emission — no orphan schedule changes without an audit trail.

### B. Lib-internal retention sweep

```text
SecurityRetentionScheduler.tick():
  policy = currentPolicy()
  await eventStore.applyRetentionPolicy(
    policy,
    initiator: AutomationInitiator('retention-policy-sweep'),
  )
        |
        v
  inside applyRetentionPolicy:
    backend.transaction((txn) async {
      truncated = await _truncateRows(txn, policy)
      purged = await _purgeRows(txn, policy)
      // existing _compacted / _purged emissions per REQ-d00138 stay as-is
      if (truncated > 0) await _emitCompacted(txn, truncated, ...)
      if (purged > 0) await _emitPurged(txn, purged, ...)
      // new: the per-sweep audit
      await _eventStore.appendInTxn(
        txn,
        entryType: kRetentionPolicyAppliedEntryType,
        entryTypeVersion: 1,
        aggregateType: 'system_retention',
        aggregateId: 'security-retention',
        eventType: 'finalized',
        data: {
          'policy_full_retention_seconds': policy.fullRetention.inSeconds,
          'policy_truncated_retention_seconds': policy.truncatedRetention.inSeconds,
          'events_truncated': truncated,
          'events_purged': purged,
          'cutoff_full': cutoffFull.toIso8601String(),
          'cutoff_purge': cutoffPurge.toIso8601String(),
        },
        initiator: AutomationInitiator('retention-policy-sweep'),
      );
    })
```

### C. Wedge recovery via `tombstoneAndRefill` (now a method)

```text
caller -> registry.tombstoneAndRefill(id, fifoRowId, initiator: UserInitiator('ops-on-call'))
  |
  v
backend.transaction((txn) async {
  // existing tombstoneAndRefill body — REQ-d00144-A..F
  result = await _tombstoneAndRefillInTxn(txn, id, fifoRowId)
  // audit emission, in same txn
  await _eventStore.appendInTxn(
    txn,
    entryType: kDestinationWedgeRecoveredEntryType,
    entryTypeVersion: 1,
    aggregateType: 'system_destination',
    aggregateId: 'destination:$id',
    eventType: 'finalized',
    data: {
      'id': id,
      'target_row_id': result.targetRowId,
      'target_event_id_range_first_seq': result.targetFirstSeq,
      'target_event_id_range_last_seq': result.targetLastSeq,
      'deleted_trail_count': result.deletedTrailCount,
      'rewound_to': result.rewoundTo,
    },
    initiator: UserInitiator('ops-on-call'),
  );
  return result;
})
```

## Error Handling

### 1. Audit emission failure → mutation rollback

Because emission lives inside the same `backend.transaction`, any `EventStore.appendInTxn` failure causes the transaction to roll back — the mutation that "happened" before the audit emission is undone. The caller sees the original `EventStore.append` exception (e.g. a hash-chain inconsistency). Better than the alternative of silently logging an unaudited mutation.

### 2. Caller forgets `initiator`

Compile-time error from Dart's `required` named parameter analysis. Same pattern as Phase 4.16's `entryTypeVersion`. No runtime path.

### 3. System entry type collision in caller's `bootstrapAppendOnlyDatastore` `entryTypes` list

If a caller passes an `EntryTypeDefinition` whose `id` matches one of the seven new reserved system entry types (or any of the existing four — `security_context_redacted/compacted/purged` plus the new seven), `bootstrapAppendOnlyDatastore` throws `ArgumentError('reserved entry type id $id cannot be supplied by caller')`. Existing logic per REQ-d00134; just extend the reserved list.

### 4. RLS interaction

System events lands in the `event_log` table just like any other event. RLS policies that apply to `event_log` (per `database/rls_policies.sql` if/when the portal-side equivalent ships) treat them as ordinary rows. No new RLS work for Phase 4.17.

### 5. Receiver-side ingest of system events

A bridged mobile→portal sync that happens to include a `system.destination_registered` event in its batch: the portal accepts it (assuming `entry_type_version >= 1` per Phase 4.16's validation) and stores it. Portal's own `DestinationRegistry` is NOT mutated — these events are *records* of mutations on the originating node, not commands to mutate elsewhere. Materialization is suppressed because the entry type's `EntryTypeDefinition.materialize == false`.

### Existing REQs amended

#### REQ-d00129 — Dynamic Destination Lifecycle

Each mutation method gains a required `Initiator initiator` parameter and a synchronous audit emission. Amend assertions A, C, F, G, H to add the `initiator` parameter to the documented signatures, and add five new assertions (J, K, L, M, N) for the audit emissions:

> **A.** `DestinationRegistry.addDestination(Destination d, {required Initiator initiator})` SHALL register `d` at any time after bootstrap; if another destination with `d.id` is already registered, the call SHALL throw `ArgumentError`. *(Rest of A unchanged.)*
>
> **C.** `DestinationRegistry.setStartDate(String id, DateTime startDate, {required Initiator initiator})` SHALL throw `StateError` if the destination already has a non-null `startDate`. *(Rest of C unchanged.)*
>
> **F.** `DestinationRegistry.setEndDate(String id, DateTime endDate, {required Initiator initiator})` SHALL return a `SetEndDateResult` enum. *(Rest of F unchanged.)*
>
> **G.** `DestinationRegistry.deactivateDestination(String id, {required Initiator initiator})` SHALL be equivalent to `setEndDate(id, DateTime.now(), initiator: initiator)`. *(Rest of G unchanged.)*
>
> **H.** `DestinationRegistry.deleteDestination(String id, {required Initiator initiator})` SHALL throw `StateError` if the destination's `allowHardDelete == false`. *(Rest of H unchanged.)*
>
> **J.** A successful `addDestination` call SHALL emit exactly one `system.destination_registered` event in the same backend transaction as the registration, with `aggregateType: 'system_destination'`, `aggregateId: 'destination:<id>'`, `eventType: 'finalized'`, `entryTypeVersion: 1`, `initiator` equal to the caller-supplied `initiator`, and `data` carrying `id`, `wire_format`, `allow_hard_delete`, `serializes_natively`, `filter_entry_types[]` (or `null`), `filter_event_types[]` (or `null`), `filter_predicate_description` (or `null`).
>
> **K.** A successful `setStartDate` call SHALL emit exactly one `system.destination_start_date_set` event in the same transaction with `aggregateId: 'destination:<id>'`, `data: { id, start_date }`, and `initiator` equal to the caller-supplied value.
>
> **L.** A successful `setEndDate` (and therefore `deactivateDestination`) call SHALL emit exactly one `system.destination_end_date_set` event in the same transaction with `aggregateId: 'destination:<id>'`, `data: { id, end_date, prior_end_date, result }`, and `initiator` equal to the caller-supplied value.
>
> **M.** A successful `deleteDestination` call SHALL emit exactly one `system.destination_deleted` event in the same transaction with `aggregateId: 'destination:<id>'`, `data: { id, allow_hard_delete }`, and `initiator` equal to the caller-supplied value.
>
> **N.** Any of the audit emissions in J, K, L, or M failing within the same transaction SHALL cause the transaction to roll back; no destination state change SHALL be observable on a node where the audit emission failed.

#### REQ-d00138 — Security Retention Policy and Redaction Audit

Add a new assertion H covering the per-sweep audit event:

> **H.** Every invocation of `EventStore.applyRetentionPolicy` SHALL emit exactly one `system.retention_policy_applied` event in the same backend transaction as the sweep, with `aggregateType: 'system_retention'`, `aggregateId: 'security-retention'`, `eventType: 'finalized'`, `entryTypeVersion: 1`, `initiator: AutomationInitiator('retention-policy-sweep')`, and `data` carrying `policy_full_retention_seconds`, `policy_truncated_retention_seconds`, `events_truncated`, `events_purged`, `cutoff_full`, `cutoff_purge`. This event SHALL be emitted regardless of whether the sweep changed any rows (zero-effect sweeps are also audited).

Existing assertions A through G (including the existing `_compacted` and `_purged` per-sweep emissions) are unchanged.

#### REQ-d00144 — tombstoneAndRefill Operation

Amend assertion A to relocate the entry point and add the `initiator` parameter:

> **A.** `DestinationRegistry.tombstoneAndRefill(String destId, String fifoRowId, {required Initiator initiator})` SHALL be the entry point for the wedge-recovery primitive (replacing the previous top-level `tombstoneAndRefill` function). It SHALL throw `ArgumentError` unless `fifoRowId` identifies the current head of the destination's FIFO. *(Rest of A unchanged.)*

Add a new assertion G:

> **G.** A successful `tombstoneAndRefill` call SHALL emit exactly one `system.destination_wedge_recovered` event in the same backend transaction as the recovery, with `aggregateType: 'system_destination'`, `aggregateId: 'destination:<destId>'`, `eventType: 'finalized'`, `entryTypeVersion: 1`, `initiator` equal to the caller-supplied value, and `data` carrying `id`, `target_row_id`, `target_event_id_range_first_seq`, `target_event_id_range_last_seq`, `deleted_trail_count`, `rewound_to`.

Existing assertions B through F are unchanged in semantics.

## Out of scope

- **Sponsor-side audit consumption.** Once the system events are in the event log and propagate over the wire (Phase 4.16's `entry_type_version` machinery handles this transparently — system entry types ride the same envelopes as application data), the portal can consume them and surface them in its operator UI. That UI is portal product spec work, not lib spec.
- **Bootstrap event.** "The lib bootstrapped at time T with config C" is interesting but outside Phase 4.17's mutation-event scope. If desired, file as a follow-up.
- **`EventStore.append`'s `dedupeByContent` flag.** Per-call config, not lifecycle. Auditing every call's flag would 10x the audit log. If a regulator asks "was dedupe ever used?", the answer is "yes" (the flag exists) — the event-by-event question is nearer to "what was the flag value when this specific event was written?" which would require stamping it on every event, not on a config-change event.
- **Sync policy changes.** `SyncPolicy` is now per-pane via Phase 4.18's hot-swap (caller controls the notifier). If the application wants to audit each policy change, it can do so in its own code without lib involvement; lib auditing of every slider drag would be excessive.
