# Phase 4.18: SyncCycle Policy Hot-Swap

**Date:** 2026-04-25
**Scope:** `apps/common-dart/event_sourcing_datastore/` (lib + tests). One example/demo cleanup follows naturally — `main.dart` can stop calling `drain` directly and use `SyncCycle.call` instead, eliminating the duplicated reentrancy guard.

## Summary

Add an optional `SyncPolicy Function()? policyResolver` parameter to `SyncCycle`. When supplied, the resolver is invoked once per cycle and its result is forwarded to `drain` as the policy for that cycle's per-destination invocations. This closes the hot-swap gap that today forces callers needing per-tick policy variation (e.g. the demo's slider) to bypass `SyncCycle` and call `drain` directly — with all the reentrancy and orchestration duplication that bypass entails.

## Architecture

```text
SyncCycle({
  required StorageBackend backend,
  required DestinationRegistry registry,
  ClockFn? clock,
  SyncPolicy? policy,                          <-- existing: captured at construction
  SyncPolicy Function()? policyResolver,       <-- new: resolved per call()
});

Mutual exclusivity: at most one of policy / policyResolver may be non-null.
Both null  -> drain falls back to SyncPolicy.defaults (today's behavior).
policy     -> drain uses captured value every cycle (today's behavior).
resolver   -> drain uses resolver() each cycle (new).
```

**Invariants:**

- The resolver, when supplied, is called exactly once per `SyncCycle.call()` invocation, before any destination's `drain` is invoked. The same resolved value is forwarded to every destination's `drain` call within that cycle.
- Existing `SyncPolicy?` constructor parameter behavior is unchanged when `policyResolver` is null. Backwards compatible for all current callers.
- The resolver type is a plain `SyncPolicy Function()` — no Flutter dependency. Callers wanting to read from a `ValueListenable<SyncPolicy>` wrap it in a one-liner: `() => notifier.value`.
- The reentrancy guard (REQ-d00125-C) and the per-destination concurrency contract (REQ-d00125-A) are unchanged. The resolver is invoked inside the guarded body, so concurrent triggers still race into the single resolved value of the winning invocation.

## Components

### 1. `SyncCycle` constructor — add the resolver parameter

In `lib/src/sync/sync_cycle.dart`:

```dart
class SyncCycle {
  SyncCycle({
    required StorageBackend backend,
    required DestinationRegistry registry,
    ClockFn? clock,
    SyncPolicy? policy,
    SyncPolicy Function()? policyResolver,
  })  : assert(
          policy == null || policyResolver == null,
          'SyncCycle: supply at most one of policy / policyResolver',
        ),
        _backend = backend,
        _registry = registry,
        _clock = clock,
        _policy = policy,
        _policyResolver = policyResolver;

  final StorageBackend _backend;
  final DestinationRegistry _registry;
  final ClockFn? _clock;
  final SyncPolicy? _policy;
  final SyncPolicy Function()? _policyResolver;
  // ... rest unchanged
}
```

### 2. `SyncCycle.call()` — invoke the resolver per cycle

```dart
Future<void> call() async {
  if (_inFlight) return;
  _inFlight = true;
  try {
    // Resolve once per cycle, after the reentrancy guard, before any drain.
    final SyncPolicy? cyclePolicy =
        _policyResolver != null ? _policyResolver() : _policy;

    final destinations = _registry.all();
    await Future.wait(
      destinations.map((d) => _drainOrSwallow(d, cyclePolicy)),
    );
    await portalInboundPoll();
  } finally {
    _inFlight = false;
  }
}

Future<void> _drainOrSwallow(Destination destination, SyncPolicy? cyclePolicy) async {
  try {
    await drain(
      destination,
      backend: _backend,
      clock: _clock,
      policy: cyclePolicy,
    );
  } catch (_) {
    // Per REQ-d00125-A — unchanged.
  }
}
```

### 3. Resolver-throws handling

If the resolver throws, the cycle aborts: no drains run, no inbound poll runs, the exception propagates out of `call()`. The reentrancy guard's `finally` clears `_inFlight` correctly. This is the right behavior — a broken policy resolver shouldn't silently fall back to defaults (which would mask a configuration bug); it should fail loud and the next trigger can retry.

### 4. Example/demo cleanup (out of strict 4.18 scope, but worth noting)

After 4.18 lands, the example's `main.dart` can replace its hand-rolled tick:

```dart
// before — current code
var syncInFlight = false;
final tick = Timer.periodic(const Duration(seconds: 1), (_) async {
  if (syncInFlight) return;
  syncInFlight = true;
  try {
    final destinations = datastore.destinations.all();
    await Future.wait(destinations.map(/* fillBatch */));
    await Future.wait(destinations.map((d) =>
        drain(d, backend: backend, policy: policyNotifier.value)));
  } catch (e, s) { stderr.writeln('[demo] sync tick error: $e\n$s'); }
  finally { syncInFlight = false; }
});
```

with:

```dart
// after — using SyncCycle with resolver
final syncCycle = SyncCycle(
  backend: backend,
  registry: datastore.destinations,
  policyResolver: () => policyNotifier.value,
);
final tick = Timer.periodic(const Duration(seconds: 1), (_) async {
  // fillBatch still happens here (SyncCycle covers drain + inbound poll only)
  final destinations = datastore.destinations.all();
  for (final dest in destinations) {
    final schedule = await datastore.destinations.scheduleOf(dest.id);
    await fillBatch(dest, backend: backend, schedule: schedule, source: source);
  }
  await syncCycle();   // includes its own reentrancy guard
});
```

The reentrancy guard duplication goes away. The drain orchestration goes through the lib's tested orchestrator. This cleanup should land in the same plan as Phase 4.18 itself; the soak test's drain-race footgun (currently mitigated by Phase 4.16's `markFinal` idempotency from `#3a`) becomes a non-issue once both panes go through `SyncCycle`.

## Data Flow

```text
trigger fires (Timer.periodic, post-append, app resume, etc.)
  |
  v
syncCycle() — single instance per pane
  |
  v
reentrancy guard: if _inFlight, return immediately   (REQ-d00125-C, unchanged)
  |
  v (guard acquired)
resolve policy:
  if _policyResolver != null  -> cyclePolicy = _policyResolver()  (NEW)
  else                         -> cyclePolicy = _policy           (unchanged)
  |
  v
for each registered destination, in parallel:        (REQ-d00125-A, unchanged)
  drain(dest, backend, clock, policy: cyclePolicy)
  |
  v
await all drains complete; one drain's throw doesn't cancel others
  |
  v
portalInboundPoll()                                   (REQ-d00125-B, unchanged)
  |
  v
release guard (try/finally)
```

The resolver being called *inside* the guard but *outside* `Future.wait` means: every destination's drain in a single cycle sees the same resolved policy value. Concurrent triggers that race into the second `call()` are turned away by the guard before resolution; they don't see a different resolved value.

## Error Handling

### 1. Both `policy` and `policyResolver` supplied

Constructor `assert` fails in debug builds. In release builds (where asserts are stripped), the resolver wins (semantic precedence: more dynamic, more explicit). Add a runtime check immediately after the assert that throws `ArgumentError` regardless of build mode — assertions are not a security boundary.

```dart
if (policy != null && policyResolver != null) {
  throw ArgumentError(
    'SyncCycle: supply at most one of policy / policyResolver',
  );
}
```

### 2. Resolver throws

Cycle aborts. No drain runs. No inbound poll. Exception propagates out of `call()`. The reentrancy guard's `finally` block correctly clears `_inFlight`, so the next trigger can attempt again with whatever the resolver does at that moment. This is preferred over silent fallback — a broken resolver should be visible.

### 3. Resolver returns null

Treated as "use `SyncPolicy.defaults`" — same as if neither `policy` nor `policyResolver` had been supplied. Lets callers express "use whatever was configured" without having to know `SyncPolicy.defaults`'s exact value.

### 4. Existing failure modes unchanged

`drain` itself still swallows per-destination failures per REQ-d00125-A. The reentrancy guard and `portalInboundPoll` placement are untouched.

## Testing

### Lib unit tests (`apps/common-dart/event_sourcing_datastore/test/`)

1. **Resolver invocation cadence** (new file: `test/sync/sync_cycle_policy_resolver_test.dart`):
   - Resolver called exactly once per `call()` invocation regardless of how many destinations are registered.
   - Resolver called *after* the reentrancy guard is acquired (so concurrent calls don't multi-invoke).
   - Resolver result forwarded as the `policy:` argument to every destination's `drain` invocation in that cycle (verify by stubbing `drain` or by stubbing the destination's `send` to inspect the policy in effect).
   - Across two sequential `call()` invocations, the resolver is called twice and may return different values (verify the second cycle uses the second resolution).

2. **Mutual exclusivity** (extend existing `sync_cycle_test.dart`):
   - Constructor with both `policy` and `policyResolver` throws `ArgumentError`.
   - Constructor with neither falls back to `SyncPolicy.defaults` (existing behavior, regression case).
   - Constructor with only `policy` works as today (regression case).
   - Constructor with only `policyResolver` invokes resolver per cycle.

3. **Resolver-throws semantics**:
   - Resolver that always throws → first `call()` throws, no drains run, `_inFlight` is cleared (a subsequent `call()` enters cleanly).

4. **Resolver-returns-null semantics**:
   - Resolver returning `null` → `drain` receives `null`, falls back to `SyncPolicy.defaults` per REQ-d00126-B (regression case).

### Example/demo updates

5. **Example `main.dart`** migrates to `SyncCycle(policyResolver: () => mobile.policyNotifier.value)` for both panes. The hand-rolled tick body collapses to just `fillBatch` + `syncCycle()`. Remove the per-pane `syncInFlight` Boolean (now covered by `SyncCycle`'s own guard).

6. **Soak test (`portal_soak_test.dart`)**: the test's hand-rolled `mobileSyncTick`/`portalSyncTick` guards added in the soak run can be removed in favor of `SyncCycle` instances. Asserts that `flutter test test/portal_soak_test.dart` still passes 60s + flush sequence within the 2-minute timeout.

7. **`portal_sync_test.dart`** and **`integration_test/dual_pane_test.dart`**: their `_Pane.tick()` helpers can either keep calling `fillBatch` + `drain` directly (test-deterministic) OR migrate to instantiating a `SyncCycle` with a resolver returning `demoDefaultSyncPolicy`. ### REQ-d00126 — SyncPolicy Injectable Value Object

Amend assertion B to add the resolver branch, and add a new assertion D:

> **B.** `drain()` and `syncCycle()` SHALL accept an optional `SyncPolicy? policy` parameter; when null, they SHALL fall back to `SyncPolicy.defaults`. `SyncCycle` SHALL additionally accept an optional `SyncPolicy Function()? policyResolver` parameter (mutually exclusive with `policy`); when supplied, the resolver SHALL be invoked exactly once per `SyncCycle.call()` invocation, after the reentrancy guard (REQ-d00125-C) is acquired and before any destination's `drain` is invoked, and the resolved value (or `SyncPolicy.defaults` when the resolver returns `null`) SHALL be forwarded to every destination's `drain` call in that cycle.
>
> **D.** Constructing a `SyncCycle` with both `policy` and `policyResolver` non-null SHALL throw `ArgumentError`. A resolver that throws SHALL cause the current cycle to abort: no drains run, `portalInboundPoll` is not invoked, the exception propagates out of `call()`, and the reentrancy guard is released so a subsequent trigger may invoke `call()` again.

Existing assertions A and C are unchanged.

## Out of scope

- **Per-destination policy.** A future requirement might want different `SyncPolicy` per destination (e.g. aggressive backoff for high-volume destinations, conservative for low). Phase 4.18 keeps a single resolved policy per cycle that applies to all destinations, matching today's `drain` parameter shape. If per-destination policy is ever needed, the cleanest extension is `SyncPolicy Function(Destination)? policyResolver` — but that's a follow-up.
- **Resolver-of-resolvers.** Composing multiple resolvers (e.g. layering "production defaults" with "ops override") is the caller's responsibility — they pass a single function that internally consults whatever sources they need.
- **`Stream<SyncPolicy>` notification of changes mid-cycle.** Within one cycle, the resolved policy is pinned. Mid-cycle policy changes don't affect the in-flight cycle. The next trigger picks up the new value. This is intentional — atomicity over freshness.
- **Resolver-aware retry inside `drain`.** `drain`'s retry/backoff logic uses the policy that was passed in at invocation. If the operator changes policy mid-retry, the *next cycle* picks it up; the in-flight retry cadence is unchanged. Phase 4.18 does not propagate policy mutations into in-flight backoff timers.
