# Phase 4.16: Per-Event Versioning (`entry_type_version` + `lib_format_version`)

**Date:** 2026-04-25
**Scope:** `apps/common-dart/event_sourcing_datastore/` (lib + tests). Example/demo updates land alongside as mechanical fixups.
**Related stubs:** Phase 4.19 (entry-type promoter chains), Phase 4.20 (datastore-format evolution mechanism), Phase 4.21 (multi-version questionnaire administration).

## Summary

Stamp two `int` version fields on every `StoredEvent`: `entry_type_version` (caller-supplied to `EventStore.append`, identifies the application schema the event was authored under) and `lib_format_version` (lib-stamped from a constant, identifies the storage shape the lib used to persist the event). Both fields are required, present from the first event written, included in the hash chain, propagated verbatim over the wire, and validated at ingest with an asymmetric `receiver_version >= event_author_version` rule that fails loud rather than silently re-stamping.

## Architecture

```text
EventStore.append(
  required int entryTypeVersion,   <-- from caller (sponsor-locked, study-locked)
  ...
)
        |
        v
StoredEvent {                            stored row JSON:
  entry_type_version: <caller-given>     "entry_type_version": N,
  lib_format_version: <lib constant>     "lib_format_version": M,
  ...                                    ...
  event_hash: SHA256(canonical JSON)     "event_hash": "..."
}
        |
        v wire (BatchEnvelope.events[]) — both fields ride along verbatim
        v
ingestBatch on receiver:
  for each event:
    if currentLibFormatVersion              < event.lib_format_version  -> IngestLibFormatVersionAhead
    if registry.byId(et).registeredVersion  < event.entry_type_version  -> IngestEntryTypeVersionAhead
    else                                                                 -> append with verbatim versions
```

**Invariants:**

- Both fields are required, non-null, on every event from day one. No silent defaults, no nullable column.
- Hash chain covers both — tampering with either field is detectable via chain-1 verification.
- Receiver preserves wire-supplied versions verbatim. Never re-stamps. Never silently rewrites.
- Asymmetric validation: `receiver_version >= event_author_version`. Newer-receiver-than-author is allowed (Phase 4.19 promoter chains will let the receiver upgrade old events on read); older-receiver-than-author is rejected loud (operator runbook: upgrade portal first, then mobile fleet).
- `EntryTypeDefinition.version: String` is deleted as part of this change — dead field today, would only mislead a future reader into thinking that's where the event-side version comes from.

## Components

All five units land in the lib (`apps/common-dart/event_sourcing_datastore/lib/`); example/demo callsite updates are mechanical fixups in the implementation plan.

### 1. `StoredEvent` — two new required `int` fields

- Add `final int entryTypeVersion;` and `final int libFormatVersion;` as constructor-required parameters.
- Add to `toMap()`/`fromMap()` JSON shape with snake_case keys `entry_type_version`, `lib_format_version`.
- `fromMap` validates presence + type — throws `FormatException('StoredEvent: missing or non-int "entry_type_version"')` (and equivalent for `lib_format_version`). Matches the existing `_requireString`/`_requireInt` validation pattern.
- Update `equals` and `hashCode` to include both fields.
- `StoredEvent.synthetic()` test factory takes both with sensible defaults (1 and 1) so test seeding stays terse.
- Add a public constant `static const int currentLibFormatVersion = 1;` on `StoredEvent` (single source of truth for the lib's current storage-format version).

### 2. `EventStore.append` — required `entryTypeVersion`; `lib_format_version` stamped from constant

- New required named parameter: `required int entryTypeVersion`.
- `append` constructs the `StoredEvent` with `entryTypeVersion: <caller's value>` and `libFormatVersion: StoredEvent.currentLibFormatVersion`.
- Hash chain (the canonical-JSON computation that produces `eventHash`) sees both fields automatically because they're in `toMap()`. No new code in the hashing path.
- `append` does NOT validate `entryTypeVersion` against `EntryTypeDefinition.registeredVersion`. Local append is the local node's prerogative; ingest is the trust boundary that validates. (See Error Handling §4 for rationale.)

### 3. `EntryTypeDefinition` — replace dead `version: String` with `registeredVersion: int`

- Delete the existing `final String version;` field, its constructor parameter, the `fromJson`/`toJson` handling for `version`, the `equals`/`hashCode` lines, the `toString` line, and the dartdoc reference.
- Add `final int registeredVersion;` as a new required constructor parameter. JSON key: `registered_version`.
- Semantics: the highest `entry_type_version` this lib build's registry will accept on ingest. Today (single-version world) it's the only value; Phase 4.21 may later expand the surface to a set or range.
- `fromJson` requires the integer; throws `FormatException` if missing or non-int.
- Greenfield project = no callers depend on the deleted field; the change is safe.

### 4. `BatchEnvelope` event payload contract — propagate both fields verbatim

- The wire schema for `events[]` items already carries arbitrary keys; both new fields naturally serialize there from `StoredEvent.toMap()` and deserialize via `StoredEvent.fromMap()`.
- Receive-side: `StoredEvent.fromMap`'s new required-field validation (§1) ensures wire payloads omitting either version are rejected with `FormatException`, caught by `ingestBatch`'s existing decode-error path → `IngestDecodeFailure`.
- No `BatchEnvelope` schema change beyond what's automatic from the per-event JSON.

### 5. `ingestBatch` — version-ahead validation + two new typed exceptions

- New exception types in `lib/src/ingest/ingest_errors.dart`:
  - `IngestLibFormatVersionAhead({required String eventId, required int wireVersion, required int receiverVersion})`
  - `IngestEntryTypeVersionAhead({required String eventId, required String entryType, required int wireVersion, required int receiverVersion})`
- Both extend the existing ingest exception hierarchy alongside `IngestDecodeFailure` / `IngestIdentityMismatch` / `IngestChainBroken`.
- Add a private validation helper inside `_ingestOneInTxn` (or its caller) that runs **before** the chain-1 verify:
  1. `if (incoming.libFormatVersion > StoredEvent.currentLibFormatVersion) throw IngestLibFormatVersionAhead(...)` (lib check first).
  2. `if (def.registeredVersion < incoming.entryTypeVersion) throw IngestEntryTypeVersionAhead(...)`.
  Where `def` is the `EntryTypeDefinition` returned by `registry.byId(incoming.entryType)`. The `def == null` case (unknown entry type) is the existing pre-version-check failure path; unchanged.
- Throwing inside the transaction rolls back the whole batch — matches existing `IngestIdentityMismatch` rollback semantics.
- Add both new exceptions to `event_sourcing_datastore.dart`'s `show` list next to the existing ingest exceptions.

## Data Flow

Two paths: local append, and wire ingest. Both stamp/validate the same two fields, with versions flowing through existing persistence + hash + transaction machinery.

```text
LOCAL APPEND (mobile, portal-local injection, system entry types)
================================================================
caller -> EventStore.append(
            entryType: 'epistaxis_event',
            entryTypeVersion: 5,           <-- caller-supplied (sponsor-locked)
            data: {...},
            ...
          )
  |
  v
EventStore stamps:
  entry_type_version = caller's value (5)
  lib_format_version = StoredEvent.currentLibFormatVersion (1)
  |
  v
Canonical-JSON hash (over toMap() — both new fields included)
  -> event_hash
  |
  v
backend.appendEvent(txn, StoredEvent{...both versions...})
  -> persisted to disk under the normal append path
  -> fillBatch/drain pick it up unchanged


WIRE INGEST (portal receives mobile's batch)
=============================================
ingestBatch(bytes, wireFormat: 'esd/batch@1')
  |
  v
BatchEnvelope.decode(bytes)
  -> envelope.events[i] is a Map<String, Object?> with
     entry_type_version + lib_format_version present
  |
  v
StoredEvent.fromMap(envelope.events[i], 0)
  -> validates required int fields; FormatException -> IngestDecodeFailure
  |
  v
Per-event version check (NEW, runs inside the existing transaction):
  if stored.libFormatVersion > StoredEvent.currentLibFormatVersion
     -> throw IngestLibFormatVersionAhead   (rolls back batch)
  def = registry.byId(stored.entryType)
  if def != null and stored.entryTypeVersion > def.registeredVersion
     -> throw IngestEntryTypeVersionAhead   (rolls back batch)
  // def == null (unknown entry type) falls through to the existing
  // pre-version-check failure path; behavior unchanged.
  |
  v
_ingestOneInTxn(txn, stored, batchContext: ...)
  -> chain-1 verify, identity check, append (existing path, unchanged)
  -> stored event now in receiver's log with verbatim wire-supplied versions
```

**Hash chain interaction.** Both fields are in `toMap()`, so they participate in the canonical JSON used for `eventHash`:

- A receiver that re-stamped either field would compute a different `eventHash` than the wire's `eventHash`; mismatch detected by chain-1 verify, throws `IngestChainBroken`. Belt-and-suspenders against accidental re-stamping.
- Tampering with either field on disk after the fact corrupts the hash chain — same protection as for `entry_type`, `aggregate_id`, etc.

**Provenance interaction.** No change. Provenance entries still record the receiver's hop/identifier/software_version. They do not need a version field of their own — the wire-author's versions are on the event itself; the receiver's lib version is implicit from the receiver's `software_version` provenance string.

**Validation ordering.** Lib-format check runs before entry-type check. Reason: an `IngestLibFormatVersionAhead` failure means the receiver fundamentally can't read the event's storage shape. Even if `fromMap` happened to succeed (because the new shape only added optional fields), we want to reject loud rather than silently accept a version we don't fully understand. Entry-type version check comes second — at that point we know the row's structural shape is compatible with this receiver's lib, and the remaining question is whether the application schema is.

## Error Handling

Three new failure modes plus two policy decisions, all using existing surfaces.

### 1. Caller forgets `entryTypeVersion` on `append`

Compile-time error from Dart's `required` named parameter analysis. No runtime path. The strongest possible enforcement.

### 2. Wire payload omits either version field

`StoredEvent.fromMap` throws `FormatException('StoredEvent: missing or non-int "entry_type_version"')` (or equivalent for `lib_format_version`). Caught by `ingestBatch`'s existing decode wrapper → `IngestDecodeFailure`. Bridge maps `IngestDecodeFailure` → `SendPermanent` (already wired). Operator-visible: mobile's Native FIFO row goes final-failed; portal's audit panel shows nothing for this event.

### 3. Wire payload supplies a version ahead of the receiver's known versions

| Condition | Exception | `SendResult` | Operator action |
| --- | --- | --- | --- |
| `event.libFormatVersion > receiver.currentLibFormatVersion` | `IngestLibFormatVersionAhead` | `SendPermanent` | Upgrade portal's lib version. |
| `event.entryTypeVersion > def.registeredVersion` | `IngestEntryTypeVersionAhead` | `SendPermanent` | Upgrade portal's entry-type registry. |
| `registry.byId(et) == null` | (existing — unchanged) | `SendPermanent` | Register the entry type. |

All three throw inside `ingestBatch`'s existing transaction, so the **whole batch rolls back** — partial ingest never observed. Matches today's `IngestIdentityMismatch` semantics.

### 4. Local append with caller-supplied version newer than the local registry's `registeredVersion`

**Allowed; the lib does NOT validate.** The lib's `append` doesn't check `entryTypeVersion` against the registry — that validation exists at ingest because it's a cross-node trust boundary. Locally, the caller is presumed to know what they're doing (sponsor-locked version is whatever the caller says it is). If the registry doesn't know that version, materializer/promoter logic will fail at materialize time, but that's a separate concern. Asymmetry is deliberate: appending is the local node's prerogative; ingest is a trust boundary.

### 5. Reserved system entry types

The lib's internal emit code calls `append(entryTypeVersion: <lib's version for that system type>, ...)`. There's no special-case path in `append`. If the lib's emit code ever passes a wrong number, that's a bug in the lib, not a runtime failure mode the caller can hit. Compile-time `required` enforces presence; correctness is a code-review concern internal to the lib.

**No silent fallbacks anywhere.** No "default to 1 if missing", no "guess from the registry if unset". Every event's versions are either supplied by an explicit decision or stamped by the lib's own constant.

### Existing REQs amended

#### REQ-d00116 — EntryTypeDefinition Schema (`spec/dev-event-sourcing-mobile.md`)

Existing assertion B today reads: *"An `EntryTypeDefinition` SHALL carry a `version` string identifying the schema version under which events of this type are written."*

The string-typed `version` field is dead in the codebase and will be replaced. Amend assertion B to:

> **B.** An `EntryTypeDefinition` SHALL carry a `registered_version` integer identifying the highest `entry_type_version` this lib build's registry will accept on `EventStore.ingestBatch`. Assertion replaces the previous `version` string assertion; the dead string field is deleted from the code path and the JSON shape.

Other REQ-d00116 assertions (A, C, D, E, F, G) are unchanged.

#### REQ-d00118 — Event Record Schema (`spec/dev-event-sourcing-mobile.md`)

Add two new assertions:

> **E.** Every event record SHALL carry a top-level `entry_type_version` integer field whose value identifies the application schema version under which the event was authored. The value is supplied by the caller of `EventStore.append` and is preserved verbatim across wire transport (REQ-d00145-B) and receiver ingest (REQ-d00145-K).
>
> **F.** Every event record SHALL carry a top-level `lib_format_version` integer field whose value identifies the storage shape the lib used to persist the event. The value is stamped by the lib at `EventStore.append` time from the constant `StoredEvent.currentLibFormatVersion`; callers of `EventStore.append` SHALL NOT supply this field.

Both new fields participate in the canonical-JSON `event_hash` computation per REQ-d00120-E (already covers "all event fields").

Other REQ-d00118 assertions (A, B, C, D) are unchanged.

#### REQ-d00141 — EventStore Append Contract (`spec/dev-event-sourcing-mobile.md`)

Existing assertion B today lists `EventStore.append`'s parameters. Amend assertion B to add `entryTypeVersion`:

> **B.** `EventStore.append({entryType, **entryTypeVersion**, aggregateId, aggregateType, eventType, data, initiator, flowToken?, metadata?, security?, checkpointReason?, changeReason?, dedupeByContent=false})` SHALL be the single public write method serving both mobile widgets and portal callers; it SHALL return the persisted `StoredEvent` or `null` when a `dedupeByContent` no-op was detected. The `entryTypeVersion` parameter is required (Dart's null-safety enforces presence at compile time); `EventStore.append` SHALL stamp the supplied value verbatim onto `StoredEvent.entry_type_version` per REQ-d00118-E.

Add a new assertion E:

> **E.** `EventStore.append` SHALL stamp `StoredEvent.lib_format_version` from the constant `StoredEvent.currentLibFormatVersion` on every event written, regardless of caller input. The constant SHALL be defined exactly once in the lib at `lib/src/storage/stored_event.dart` and SHALL identify the storage shape the current lib build produces.

Add a new assertion F:

> **F.** `EventStore.append` SHALL NOT validate the caller-supplied `entryTypeVersion` against `EntryTypeDefinition.registered_version` of the local registry. Local append is the local node's prerogative; cross-node validation is performed at ingest per REQ-d00145.

Other REQ-d00141 assertions (A, C, D) are unchanged.

#### REQ-d00145 — EventStore Ingest Contract (`spec/dev-event-sourcing-mobile.md`)

Add two new assertions covering the asymmetric version-ahead validation:

> **L.** Before chain-1 verify (REQ-d00145-C), `ingestBatch` SHALL evaluate `incoming.lib_format_version > StoredEvent.currentLibFormatVersion`; on true, SHALL raise `IngestLibFormatVersionAhead` carrying the offending `event_id`, `wire_version`, and `receiver_version`, rolling back the entire batch (the all-or-nothing semantics of REQ-d00145-A apply).
>
> **M.** After the lib-format check (REQ-d00145-L) and before chain-1 verify (REQ-d00145-C), `ingestBatch` SHALL look up `def = registry.byId(incoming.entry_type)`; when `def != null` and `incoming.entry_type_version > def.registered_version`, SHALL raise `IngestEntryTypeVersionAhead` carrying the offending `event_id`, `entry_type`, `wire_version`, and `receiver_version`, rolling back the entire batch. When `def == null` (unknown entry type), the existing pre-version-check failure path is unchanged.

Amend existing assertion K to enumerate the two new fields among the originator-identity fields the receiver SHALL NOT mutate:

> **K.** `ingestBatch` and `ingestEvent` SHALL NOT mutate any originator identity field other than `sequence_number` (`event_id`, `aggregate_id`, `aggregate_type`, `entry_type`, **`entry_type_version`, `lib_format_version`,** `event_type`, `data`, `initiator`, `flow_token`, `client_timestamp`, `previous_event_hash`) on the incoming event. *(Rest of assertion K unchanged.)*

Existing wire-format assertion B (`batch_format_version` = `"1"`) is unchanged — both new per-event fields ride inside `events[i]` payloads, not at the envelope level.

## Out of scope

- **Promoter chains and version-aware materialization.** When a receiver registers `registeredVersion: 5` and ingests an event with `entry_type_version: 3`, the receiver accepts and stores the event verbatim — but the materializer rebuild may not interpret v=3 events the same way as v=5 events. Designing the per-`(entryType, fromVersion, toVersion)` promoter callback registry is **Phase 4.19**.
- **Multi-version registry shape.** `EntryTypeDefinition.registeredVersion` is a single `int` in 4.16. The multi-version-per-mobile-install reality (different sponsors locked at different versions of the same entry-type) may require expanding to a `Set<int>` or similar — **Phase 4.21**.
- **Lib-format upgrade-on-read mechanism.** When a future lib release bumps `currentLibFormatVersion` from 1 to 2, old events on disk (stamped `lib_format_version: 1`) need an upgrade path — lazy, eager, or lockstep. The field exists from day one (this phase); the evolution mechanism is **Phase 4.20**.
- **Wire-format negotiation / backwards compatibility.** `BatchEnvelope.batch_format_version` already exists with reject-on-mismatch behavior. Designing a richer compat story is **Phase 4.20**.
