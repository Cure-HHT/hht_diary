# Read-Side API Gaps (Phase 4.11 Design Spec)

**Date**: 2026-04-24
**Ticket**: CUR-1154 (continuation of the `mobile-event-sourcing-refactor` branch)
**Phase**: 4.11 — follows the Phase 4.9 / 4.10 sync-through work. Library-only, additive.
**Scope**: `event_sourcing_datastore` public API. No consumer code changes. No storage schema changes.
**Companion**: deficits 1.4–1.5 (reactive / streaming layer) are carved out into the Phase 4.12 stub — see `2026-04-24-phase4.12-reactive-read-layer-stub.md`.

## 1. Problem

An example-app audit conducted 2026-04-24 (late Phase 4.9 / pre-4.10) surfaced a consistent pattern: every read-side UI panel polls with `Timer.periodic(500ms)`, does full-log scans each tick, and — where the public API falls short — reaches past `StorageBackend` into the sembast store directly. Six concrete deficits, ranked by severity:

### 1.1 No `findEventById`
Anything that resolves "I have an event id, give me its sequence number / metadata" is O(N) over the event log. `fifo_panel.dart` today fetches `findAllEvents(limit: 500)` every 500 ms per destination to build an `eventId → sequenceNumber` map purely to label each FIFO row with its latest event's sequence number. `StorageBackend` has `findEventsForAggregate` and `findAllEvents` but no by-id lookup.

### 1.2 No `listFifoEntries(destinationId, …)`
`StorageBackend` exposes point/head/summary queries (`readFifoHead`, `readFifoRow(dest, entryId)`, `wedgedFifos`, `anyFifoWedged`) but no "enumerate the queue." Consumers that want a queue-inspector view open the raw sembast store by name:

```dart
final store = intMapStoreFactory.store('fifo_${destinationId}');
final records = await store.find(backend.debugDatabase());
```

(`fifo_panel.dart:76–80`.) That couples the caller to an undocumented sembast layout, the `fifo_<id>` naming convention, and the `debugDatabase()` escape hatch.

### 1.3 `debugDatabase()` is load-bearing in non-debug code
`sembast_security_context_store.dart:28, :126` uses it in the security store. The method name says "debug-only"; the usage says "we didn't have a typed API so we leaked the handle." Anyone reading the security store learns that reaching for `debugDatabase()` is acceptable, and the pattern spreads.

### 1.4 `detail_panel.dart` uses `findAllEvents(limit: 100000)` every 500 ms
The `100000` is a "please don't page me" cap, not a real bound. The API accepts `afterSequence` but no polling consumer uses it, because there's no observable "new events since X" primitive to drive an incremental poll against.

### 1.5 No reactive / changefeed API
`StorageBackend` exposes only `Future<List<…>>`. Every view is either stale between polls or polled at UI tick rate. N open panels × 2 Hz × full-table scans scale linearly with log size and panel count. At scale a production client forks into the same pattern or re-implements it.

### 1.6 No coordination primitive across consumers
N FIFO panels × 2 Hz × independent `scheduleOf` + `findAllEvents` reads, no shared cache. Example-app sloppiness, but also a signal: the library provides no coordination point consumers can build on.

## 2. Scope decisions

**In scope for Phase 4.11** (narrow, additive, unblocks mobile read paths):

- §3.1 `findEventById` — closes deficit 1.1.
- §3.2 `listFifoEntries` — closes deficit 1.2.
- §3.3 `debugDatabase` audit + removal from non-debug call sites — closes deficit 1.3. Security-context store gets a typed API surface; example app's FIFO reach-around is eliminated by §3.2.
- §3.4 example-app migration — `fifo_panel.dart` uses §3.1 + §3.2; security store uses the new typed surface. Proves the new APIs carry the intended load.

**Deferred (NOT in Phase 4.11)**:

- Reactive / streaming APIs (deficits 1.4, 1.5). Adds a new concurrency contract, a change-notification mechanism, and backing storage instrumentation. Carved into the **Phase 4.12 stub** (`2026-04-24-phase4.12-reactive-read-layer-stub.md`) for a later brainstorm. `findAllEvents(afterSequence:)` already supports incremental polls; consumers that care can start using it today without a new API.
- Shared-cache / coordination primitive (deficit 1.6). Cross-cuts the UI tier; not obviously library-layer work. Folded into the Phase 4.12 stub for re-evaluation alongside the reactive-layer design.

Rationale for this split: Phase 4.11 is pure API additions over existing storage with existing tests, no new concurrency contract, reviewable in one pass. The reactive layer is a separate design problem and pulling it into 4.11 would blow up the PR.

## 3. Design

### 3.1 `findEventById(String eventId) → Future<StoredEvent?>`

Add to `StorageBackend`:

```dart
/// The event with [eventId], or null when no such event exists in the
/// log. Non-transactional, read-only. O(1) under the sembast implementation
/// (indexed lookup on `event_id`); other implementations SHALL provide an
/// equivalent single-row lookup rather than a full-table scan.
Future<StoredEvent?> findEventById(String eventId);
```

**Sembast backing**: the event store is keyed on `sequence_number` and indexed on `event_id` via a sembast `Finder(filter: Filter.equals('event_id', id))`. If no index exists today, Phase 4.11 adds one as part of the task that wires up this method. (An unindexed `Filter.equals` in sembast still works but degrades to a scan — the REQ requires indexed-lookup complexity.)

**Why on `StorageBackend`, not `EventStore` / `EntryService`**: the lookup is storage-shaped, not service-shaped, and the sembast index belongs to the storage layer. Callers that want "event id → metadata" (UI, verification, hash-chain walkers from Phase 4.9) hit the storage backend directly.

**Null semantics**: a not-found event returns `null`. Callers are expected to distinguish "id is malformed" (never valid) from "event was never appended" (may appear later after ingest) at their own layer.

### 3.2 `listFifoEntries(String destinationId, {int? afterSequenceInQueue, int? limit}) → Future<List<FifoEntry>>`

Add to `StorageBackend`:

```dart
/// All FIFO entries for [destinationId], ordered by `sequence_in_queue`
/// ascending. Optionally sliced by [afterSequenceInQueue] (exclusive) and
/// [limit] so callers can page through a large queue without materializing
/// it entirely.
///
/// Returns `FifoEntry` objects, not raw maps — callers SHALL NOT reach
/// around this method to `intMapStoreFactory.store('fifo_<id>')` for the
/// same data. The `fifo_<id>` store name is an implementation detail of
/// the sembast backend and is not part of the public contract.
Future<List<FifoEntry>> listFifoEntries(
  String destinationId, {
  int? afterSequenceInQueue,
  int? limit,
});
```

**Ordering**: ascending by `sequence_in_queue`. UI callers that want most-recent-first reverse in the view layer (same as `event_stream_panel.dart` does today with events).

**Pagination**: `afterSequenceInQueue` is exclusive, matching `findAllEvents`'s `afterSequence` convention. `limit` is optional; a null limit returns the whole queue (acceptable for mobile-scale FIFOs; the pagination affordance is there for the reach-around scenario and for future ops dashboards against multi-day queues).

**Return type**: `List<FifoEntry>`, not raw sembast records. Example-app code today reads raw `Map<String, Object?>` and string-keys into it (`row['event_ids']`, `row['sequence_in_queue']`, etc.) — the new API returns typed objects and the example migrates off raw maps as part of §3.4.

### 3.3 `debugDatabase()` — audit and narrow

Two non-example call sites exist today:
- `lib/src/security/sembast_security_context_store.dart:28` (read)
- `lib/src/security/sembast_security_context_store.dart:126` (write)

Both open named sembast stores by string and do direct finds/puts for security-context rows. The fix has two options:

**Option A (preferred)**: add typed methods to `StorageBackend` for the two operations the security store performs (`readSecurityContext(principalId)`, `writeSecurityContextTxn(txn, context)` or whatever the read/write shapes are), backed by the same sembast store internally. Security store stops calling `debugDatabase()`. Document `debugDatabase()` as "example/test only."

**Option B**: rename `debugDatabase()` to something like `unsafeRawDatabase()` and leave the security store as-is. Cheaper but less satisfying — the security store is still bypassing the public contract and the name change is cosmetic.

This design spec picks Option A. The two methods added are storage-shaped, short, and match the existing `readFillCursor` / `writeFillCursor` pattern. The example app's reach-around (§3.2) is independently closed by `listFifoEntries`, so after Phase 4.11 `debugDatabase()` has zero non-test call sites in the library tree.

Exact method shapes pending Task 2 (brief read of the current security-store code to mirror the types correctly).

### 3.4 Example-app migration

`apps/common-dart/event_sourcing_datastore/example/lib/widgets/fifo_panel.dart`:

- Replace the sembast store open + `find` (lines 75–80) with `backend.listFifoEntries(destinationId)`.
- Replace the `findAllEvents(limit: 500)` + `eventId → sequenceNumber` map build (added in the most recent change) with a per-row `findEventById(eventIds.last)` call inside `_rowList`'s builder. At O(1) per tile this is cheaper than the 500-row scan and eliminates the state field `_seqByEventId`.
- `_rows` becomes `List<FifoEntry>` instead of `List<Map<String, Object?>>`; the tile accesses typed fields (`row.sequenceInQueue`, `row.eventIds`, `row.finalStatus`, `row.attempts`, `row.entryId`) instead of string keys.

Other panels (`detail_panel.dart`'s `findAllEvents(limit: 100000)`, `event_stream_panel.dart`'s 500 ms poll) are explicitly out of scope for 4.11 — they fall under deficit 1.5 (reactive layer) and are tracked in the Phase 4.12 stub.

## 4. REQ impact

Two new REQ-d numbers claimed. Current ceiling at spec time is REQ-d00144. Phase 4.9 claims **REQ-d00145** (`EventStore.ingestBatch` / `ingestEvent` contracts) and **REQ-d00146** (`verifyEventChain` / `verifyIngestChain` contracts), and extends REQ-d00115 + REQ-d00120 without new numbers. Phase 4.10 extends REQ-d00128 without claiming a new number. After Phase 4.9 and Phase 4.10 land, the ceiling is REQ-d00146. Phase 4.11 therefore claims:

- **REQ-d00147**: `StorageBackend.findEventById` contract — indexed single-row lookup by `event_id`, returns null when absent, O(1) complexity requirement on the sembast backend.
- **REQ-d00148**: `StorageBackend.listFifoEntries` contract — ordered enumeration of a destination's queue, pagination via `afterSequenceInQueue` / `limit`, returns typed `FifoEntry` objects. Includes the prohibition on callers opening the `fifo_<id>` store directly.

Task 2 re-checks the ceiling before committing.

No existing REQ changes. `debugDatabase()` is not specified in REQ-land today (it's a `StorageBackend` method without a corresponding REQ); the narrowing in §3.3 tightens its doc comment to "example / test only" without REQ churn.

`spec/INDEX.md` gains two new REQ entries; pre-commit hook regenerates the hash column for `spec/dev-event-sourcing-mobile.md`.

## 7. Risks

### Risk 1: `findEventById` without an index silently O(N)
An implementation that forgets to add the sembast index on `event_id` still passes functional tests (lookups return the right event) but has wrong complexity. Mitigation: the REQ explicitly specifies indexed lookup; the sembast task in the plan adds the index as a visible step; a test inspects the sembast database configuration (or uses sembast's query-plan hook, if available) to assert an index exists. If no inspection hook exists, document the requirement in the sembast backend's implementation comment and rely on code review.

### Risk 2: `listFifoEntries` return shape churn
Returning `List<FifoEntry>` rather than raw maps means example-app and any downstream consumer touches the shape. In-repo this is a one-file change (§3.4). Out-of-repo consumers don't exist yet (mobile hasn't cut over). Low risk.

### Risk 3: Security-store migration under Option A
The security store is a sensitive file — any refactor risks subtle bugs. Mitigation: §3.3's added methods are near-mechanical translations of the existing `debugDatabase()` call sites (open the same store, same finder, same put). The security store's own tests should pass unchanged; if they rely on the internal store layout via side channels, that's a separate concern to surface in Task 1's code read.

### Risk 4: Scope creep into reactive layer
Reviewer feedback may push "while you're in there, add a Stream." Hold the line — reactive is deferred per §2. 4.11 is additive read primitives only.

## 8. Out of scope / not changed

- Reactive / streaming APIs. Deferred to a later phase.
- Shared-cache primitive. Deferred; re-evaluate after the reactive-layer design.
- `detail_panel.dart` and `event_stream_panel.dart` continue to poll with their current shapes. Their migration belongs to the reactive-layer phase.
- FIFO storage schema. `FifoEntry` fields unchanged; only the enumeration API is added.
- `EntryService`, `EventStore`, materializer, sync machinery — all unchanged.
- Consumer apps (daily-diary, edc, etc.) — unchanged; they don't use these APIs today.
