# Phase 4.21: Multi-Version Questionnaire Administration (stub)

**Date:** 2026-04-25
**Status:** Stub — full design deferred. App-level architecture (not strictly within the lib's scope), but captured here so the next architect inherits a bounded design space rather than discovering the constraints by accident.
**Requirements:** deferred until full design — REQs will be claimed during Phase 4.21's brainstorm + spec, which may register REQs against `spec/dev-questionnaire.md` rather than `spec/dev-event-sourcing-mobile.md` given the app-level concern.

## Why this is a stub, and why it matters

The lib (`event_sourcing_datastore`) records what callers tell it to record. It deliberately doesn't enforce "one current version per entry_type" — see Phase 4.16, where `entry_type_version` is caller-supplied with a registry default. That's the right boundary for the lib.

But there's a level above the lib — the unified mobile app and its sponsor/study/patient context — that owns the question of *which version to administer for a given patient at a given time*. That layer has hard, non-obvious constraints, and the wrong choice for it would push problems back down into the lib (e.g., "actually, please make the registry sponsor-aware after all"). Phase 4.21 bookmarks that design space.

## Constraints (the unmovable parts)

- **One mobile binary serves multiple sponsors.** A single mobile install must support every sponsor whose patients it might serve.
- **Per-study version locks.** When a sponsor approves a study, the questionnaire versions that study uses are locked. Changing a locked version requires sponsor approval — onerous, granted only for very good reasons (e.g., a regulatory finding).
- **Concurrent versions of the same entry-type.** Sponsor A's study is locked at `epistaxis_event v=3`; Sponsor B starts a new study and picks `epistaxis_event v=5`. The mobile app must administer both correctly to the right patients without cross-contamination.
- **Audit completeness rule (project-wide):** events should be self-contained for an auditor — record the question text the patient saw, the answer choices presented, the chosen answer, and the canonical/translated answer if applicable. The auditor must not have to dereference an external "questionnaire definition v3.json" to understand the data.

## Two candidate architectures

### A. Versions as separate entry_types

`epistaxis_event_v3`, `epistaxis_event_v5`, etc. are registered as fully distinct `EntryTypeDefinition`s. The `entry_type_version` field always equals 1 (or is unused).

- Pros: no version-promoter logic ever needed; the registry's existing `byId(entryType)` lookup works unchanged; sponsor lock is encoded in *which entry_type id* the study selects.
- Cons: registry grows linearly with versions; queries like "all epistaxis events for this patient" must enumerate types; the mobile app's widget registry duplicates UI per version (mitigated only if widgets share a base widget).

### B. Versioned entry-type with caller-supplied version

`epistaxis_event` is one entry_type; `entry_type_version` is per-event (3 for Sponsor A's events, 5 for Sponsor B's). Registry stores all known versions; the questionnaire administration code looks up the study's locked version per administration and supplies it.

- Pros: registry stays one-entry-per-type; "all epistaxis events" is one query; promoter chains (Phase 4.19) can normalize across versions for cross-study analysis.
- Cons: the registry needs to expose multiple versions per entry_type (a shape change beyond Phase 4.16's `defaultVersion`); widgets must dispatch on `entry_type_version` to render correctly; cross-study aggregation needs explicit handling.

### C. Hybrid — A for "custom" types, B with data-driven payload for "generic" types

Some questionnaires are heavily custom (epistaxis-specific logic, custom widgets) — those use approach (A): `epistaxis_event_v3` is one entry_type, `epistaxis_event_v5` is another. Re-author the widget per version when needed.

Other questionnaires fit a generic class (e.g. `Class3 = N pages of prelude text + M multi-choice questions with 4 choices each`). For these, the entry_type is `generic_questionnaire_class3`, and the actual content (questions, choices, translations) is downloaded from the portal at administration time. The event records a snapshot of the rendered questionnaire content alongside the answers — the audit payload is self-contained.

- Pros: bounded widget count; portal-controlled content for generic types; lighter mobile updates for the common case (new generic-class questionnaire = portal-side data change only); custom types still get the rigor they need.
- Cons: requires designing the "questionnaire class" taxonomy; requires a portal-to-mobile definition push or pull mechanism; the mobile app must persist downloaded definitions correctly; more moving parts overall.

## Open design questions to resolve in Phase 4.21's brainstorm

- Where do study version locks live? (Portal-side per-study config? Pushed to mobile at enrollment? Cached on mobile?)
- How does mobile know which version/class to administer for a given patient? (Patient → Study → Locked questionnaire set → Per-questionnaire version.)
- Does the portal *push* questionnaire definitions to mobile, or does mobile *pull* them on demand?
- What's the offline story? A mobile that hasn't synced recently must still be able to administer the right questionnaire to enrolled patients.
- For approach (C), what's the questionnaire-class taxonomy (how many classes? what shapes do they cover? when does a new questionnaire need a new class)?
- How does the audit-completeness rule interact with translations? Record original-language text + translated text + chosen answer + canonical-language equivalent of the chosen answer?
- Where does the line between "lib concern" and "app concern" land? E.g., does the lib provide a "questionnaire definition" entity, or is that purely app code? (Default answer: app code; the lib only knows about events.)

## Anchor points

- Phase 4.16 design: `entry_type_version: int` is caller-supplied with optional registry default. The lib does NOT pick the version on the caller's behalf for application data.
- Phase 4.19 design: per-`(entry_type, fromVersion, toVersion)` promoter callbacks. Promoters operate on event payload shape; they cannot reach across entry_type boundaries.
- Project memory `project_single_tenant_per_sponsor.md`: portal is single-tenant per sponsor (own VPC). Version-lock state lives per-sponsor on the portal. Mobile is the multi-tenant carrier.
- Project memory `project_fda_requirement_architecture.md`: three-layer FDA requirement structure. Questionnaire definitions and their versioning likely intersect with the foundation layer.

## Out of scope for 4.21

- The lib's `entry_type_version` field semantics — already designed in 4.16.
- The promoter chain mechanism — designed in 4.19.
- The lib-internal format versioning — designed in 4.20.
- Sponsor-side study management workflow on the portal (separate concern, lives in portal product spec).
- Translation/localization infrastructure (touches the audit-completeness rule but is its own design).
