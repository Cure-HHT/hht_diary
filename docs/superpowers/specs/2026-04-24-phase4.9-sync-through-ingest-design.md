# Sync-Through Ingest (Phase 4.9 Design Spec)

**Date**: 2026-04-24
**Ticket**: CUR-1154 (continuation of the `mobile-event-sourcing-refactor` branch)
**Phase**: 4.9 — follows Phase 4.8 (merge materialization). Library-only, additive.
**Scope**: `event_sourcing_datastore` library — new `EventStore.ingestBatch` / `ingestEvent` / `logRejectedBatch` / verification APIs; canonical batch-format definition; `ProvenanceEntry` schema extensions; REQ changes. No consumer apps are touched; portal HTTP transport remains deferred. Outgoing-side native-format optimization carves off into Phase 4.13.

## 1. Problem

The library's sole write path today is `EventStore.append` (REQ-d00141), which mints fresh identity for every record: new `event_id`, next `sequence_number`, fresh `event_hash`, local `initiator`, local `provenance[0]`. That shape is correct for events **originated locally** — one mobile device writing its own diary events into its own log.

The moment an event travels between systems — mobile sends to the diary server, diary server forwards to sponsor portal, any future hop relays to another store — a second kind of write is needed: **receiver-side append** that preserves the incoming event's origin identity verbatim, records the receipt as a new hop in the event's provenance chain, and integrates the event into the receiver's own tamper-evident log. The library has no such path today.

Two audit requirements frame the design:

**Requirement 1 — cross-store matching.** An auditor with access to both the originator's datastore and a destination's datastore must be able to, record-for-record, match the destination's copy of an event against the originator's copy and verify that the originator's identity transited intact. The join key is `event_id`; the proof is that a walk of the destination's copy reconstructs the originator's `event_hash` byte-for-byte.

**Requirement 2 — destination-local tamper-evidence across originators.** A destination that ingests from many originators (multiple mobile device installations, multiple FIFOs within a single installation, future additional sources) must be able to prove that its *own* event log — the full interleaved sequence of receipts — has not been tampered with, independently of any originator's chain. No event inserted, removed, or reordered.

These two requirements are orthogonal. They describe two distinct tamper-evident chains over the same set of records.

## 2. Design

### 2.1 Two hash chains

**Chain 1 (per-event, spans hops).** An event's identity travels from origin through N receivers. Each receiver hop records what it observed on receipt; walking the chain backward recovers the origin-side `event_hash` and verifies identity preservation. Mechanism: `arrival_hash` on each receiver's provenance entry. Requirement 1 is satisfied by an auditor computing Chain 1 on the destination's copy and comparing the recovered origin hash to the originator's stored `event_hash`.

**Chain 2 (per-destination, spans events).** A destination's event log is a sequence of receipts — subject events ingested from upstream, and receiver-originated audit events emitted during ingest (see §2.9). Each event in the sequence links to the prior event in the same sequence regardless of which originator it came from. Mechanism: `previous_ingest_hash` on each event's receiver-hop provenance entry, pointing to the destination-local `event_hash` of the event immediately preceding it in ingest order. Requirement 2 is satisfied by walking Chain 2 and verifying each link.

The two chains are stored in the same place — a receiver hop's entry in `metadata.provenance`. Fields are orthogonal: `arrival_hash` serves Chain 1; `previous_ingest_hash` + `ingest_sequence_number` serve Chain 2. A single provenance entry carries both.

### 2.2 `ProvenanceEntry` schema additions (extends REQ-d00115)

Four new nullable fields; all null on an origin entry (the `provenance[0]` entry stamped by the originator):

- **`arrival_hash: String?`** — the `event_hash` observed on the wire at the moment this hop received the event, **before** this hop recomputes its own hash. Equivalently, the `event_hash` value the previous hop stored. Null on origin; non-null on every receiver entry.
- **`previous_ingest_hash: String?`** — the destination-local `event_hash` of the most-recent event in this destination's log at the moment this event was ingested at this hop. Null on origin; null on the first-ever event ingested at a receiver (no predecessor); non-null thereafter.
- **`ingest_sequence_number: int?`** — destination-local monotone counter at ingest time, used as the Chain 2 ordering index. Null on origin; non-null on every receiver entry. Monotone across all events ingested at a given receiver, regardless of originator.
- **`batch_context: BatchContext?`** — populated when this hop received the event as a member of a `ingestBatch` call; null on process-local `ingestEvent` calls and on origin entries. Shape defined in §2.3.

Existing fields (`hop`, `received_at`, `identifier`, `software_version`, `transform_version`) are unchanged.

### 2.3 Canonical batch format and `BatchContext`

Phase 4.9 defines a single canonical batch envelope — format identifier `"esd/batch@1"` — that `ingestBatch` accepts:

```json
{
  "batch_format_version": "1",
  "batch_id": "<uuid>",
  "sender_hop": "<hop_id>",
  "sender_identifier": "<device_id or server_id>",
  "sender_software_version": "<package-name@semver>",
  "sent_at": "<ISO 8601 with timezone>",
  "events": [<StoredEvent JSON>, <StoredEvent JSON>, ...]
}
```

Wire bytes SHALL be the UTF-8 rendering of RFC 8785 (JCS) canonical JSON of this object. Each entry in `events` is a StoredEvent JSON per REQ-d00118, carrying that event's `event_hash` and `metadata.provenance` as produced by the immediately-preceding hop.

Callers whose wire transport uses a different format (sponsor-specific CSV, Rave EDC XML, etc.) are responsible for converting to `esd/batch@1` bytes before calling `ingestBatch`, or for registering a pluggable `BatchCodec` — pluggable codecs are deferred; Phase 4.9 supports exactly one format.

**`BatchContext`** — the per-event record of batch membership. Stamped into each subject event's receiver-hop `ProvenanceEntry`:

```dart
class BatchContext {
  final String batchId;               // from batch envelope
  final int batchPosition;            // 0-indexed position in events[]
  final int batchSize;                // length of events[]
  final String batchWireBytesHash;    // sha256 hex of the received canonical bytes
  final String batchWireFormat;       // "esd/batch@1"
}
```

All five fields together recover the context an auditor needs: "which batch, what position, how large, what bytes-hash, which format." An auditor holding claimed wire bytes B can `sha256(B)` and join against `batchWireBytesHash` across events to prove the bytes correspond to a specific ingest. Reconstruction of the wire bytes from stored events is deterministic (strip each event's hop provenance entry to its arrival state, sort by `batchPosition`, wrap in the envelope with the shared metadata drawn from each event's `BatchContext`, JCS-canonicalize).

Receiver-originated events (duplicate-received, batch-rejected — §2.9) ALSO carry a `BatchContext` in their `provenance[0]` when they were emitted in response to a batch, so query-by-`batchId` recovers every event the batch produced (subject ingests + any audit markers).

### 2.4 Hash recompute on ingest

`event_hash` is the SHA-256 over JCS-canonicalized identity fields (REQ-d00120). Because `metadata.provenance` is part of the hashed identity, **appending a receiver provenance entry changes the hash**. The receiver recomputes `event_hash` after stamping its own entry; the new value covers `identity + metadata.provenance[0..N]` where N is this receiver's index. The stored `event_hash` on the receiver's copy is therefore "current-hop's output hash," not the originator's.

Cross-store byte-for-byte comparison of raw `event_hash` would fail — the originator's copy hashes `identity + provenance[0]` while the destination's copy hashes `identity + provenance[0..1]`. Requirement 1 is met not by comparing raw `event_hash` but by walking Chain 1 on the destination's copy: each hop's `arrival_hash` equals the hash the prior state would produce, and the walk bottoms out at origin (where `provenance[0].arrival_hash` is null and `event_hash` equals `hash(identity + provenance[0])`).

Originator identity fields are **never** rewritten by a receiver: `event_id`, `aggregate_id`, `aggregate_type`, `entry_type`, `event_type`, `sequence_number`, `data`, `initiator`, `flow_token`, `client_timestamp`, `previous_event_hash` all pass through verbatim. Only `metadata.provenance` grows (receiver appends its entry), and the derived `event_hash` is recomputed.

### 2.5 `ingestBatch` — wire-side entry point

Signature:
```dart
Future<IngestBatchResult> ingestBatch(
  Uint8List bytes, {
  required String wireFormat,
});
```

**On success**: N events (one per subject in the batch) are stored, each with its receiver-hop `ProvenanceEntry` carrying all four of `arrival_hash`, `previous_ingest_hash`, `ingest_sequence_number`, and `batch_context`. Duplicates (§2.8) produce additional `ingest.duplicate_received` events in the ingest-audit aggregate but do not increase the count of stored subject events. No separate "batch received" event is emitted on success — the per-event `batch_context` is the audit record.

**On any failure** (decode error, subject-event Chain 1 mismatch, identity mismatch on a known event id, backend storage error): the entire transaction rolls back — no events from the batch land — and the method throws a typed exception (`IngestDecodeFailure`, `IngestChainBroken`, `IngestIdentityMismatch`, or a wrapped backend exception). The library does **not** emit a rejection audit event internally. Callers that want forensic retention of the rejected bytes invoke `logRejectedBatch` (§2.7) after catching the exception.

Flow (within one `StorageBackend.transaction`):

1. Compute `wire_bytes_hash = sha256(bytes)`. Parse bytes per `wireFormat` into a batch envelope + `List<StoredEvent>`. Decode failures throw `IngestDecodeFailure` immediately (no transaction opened yet).
2. Open the transaction.
3. For each decoded subject event, in order:
   - **Chain 1 verify** on the incoming `metadata.provenance`. Failure throws `IngestChainBroken`.
   - **Idempotency check** by `event_id` (§2.8):
     - Known, hash-matching stored copy → emit `ingest.duplicate_received` (§2.9) with `BatchContext` populated; continue.
     - Known, hash-differing → throw `IngestIdentityMismatch`.
     - Unknown → proceed.
   - Build receiver `ProvenanceEntry`: `arrival_hash` = the wire `event_hash` (pre-recompute), `previous_ingest_hash` = current Chain 2 tail, `ingest_sequence_number` = counter + 1, `batch_context` = the shared BatchContext for this batch.
   - Append to `metadata.provenance`. Recompute `event_hash`. Persist via a storage-backend method that keys by `ingest_sequence_number` (§2.12). Advance counter and Chain 2 tail.
4. Commit. Return `IngestBatchResult`.

Any throw inside step 2–4 aborts the transaction (rollback). Caller sees the exception; no side effects in the event log.

### 2.6 `ingestEvent` — process-local entry point

Signature:
```dart
Future<PerEventIngestOutcome> ingestEvent(StoredEvent incoming);
```

Same per-event logic as step 3 of `ingestBatch`, minus the batch wrapper. Stamped receiver `ProvenanceEntry` has `batch_context = null` (no batch association). Used by tests and future library-internal sync-through flows (e.g., an intra-process transform that stores the transformed event as an ingest at a secondary store).

Publicly documented as "in-process callers only; wire-boundary callers use `ingestBatch`." A caller that bypasses `ingestBatch` for wire-level work has no batch-level audit trail; the library does not attempt to detect this — it is a caller-layer discipline.

`ingestEvent` opens and owns its own transaction. Idempotency is identical to §2.8 (duplicate emits `ingest.duplicate_received` in the ingest-audit aggregate, but with `batch_context = null`; identity mismatch throws). On any failure, the method throws — no internal rejection-audit emission. Process-local failures are programming errors; wrapping them in a store-side audit event would be noise.

### 2.7 `logRejectedBatch` — caller-composed rejection audit

Signature:
```dart
Future<void> logRejectedBatch(
  Uint8List bytes, {
  required String wireFormat,
  required String reason,       // enum: chainBroken, identityMismatch, decodeFailure, backend, ...
  String? failedEventId,
  String? errorDetail,
});
```

Emits a single `ingest.batch_rejected` system event (§2.9) under the ingest-audit aggregate. The event's `data` carries the supplied bytes verbatim plus the caller-supplied metadata. Opens its own transaction; stamps Chain 2 normally; does not interact with `ingestBatch` state.

Used by wire-layer callers after `ingestBatch` throws, when the deployment wants a forensic record of the rejected bytes in the event store. Example composition:

```dart
try {
  final result = await store.ingestBatch(bytes, wireFormat: 'esd/batch@1');
  // success — respond 200
} catch (e) {
  // Best-effort audit; do not let audit failure mask the ingest failure.
  try {
    await store.logRejectedBatch(
      bytes,
      wireFormat: 'esd/batch@1',
      reason: _classify(e),
      failedEventId: _extractEventId(e),
      errorDetail: e.toString(),
    );
  } catch (_) { /* surface to transport-level log; do not rethrow */ }
  // respond 4xx / 5xx
}
```

Callers in debug/test contexts may choose not to call `logRejectedBatch`; the library makes no promise that every rejected batch has a store-side record. Audit-responsibility at the library layer is opt-in.

The rejected-batch event is emitted unconditionally — `logRejectedBatch` does not verify whether the supplied bytes were actually rejected, nor does it refuse to emit a second rejected-batch event for the same `wire_bytes_hash`. Callers that want dedup-on-rejection implement it at their own layer.

### 2.8 Idempotency model

A subject event's `event_id` is the idempotency key for `ingestBatch` / `ingestEvent`. Three outcomes:

| Condition | Response |
| --- | --- |
| `event_id` not in local store | Ingest normally: stamp receiver provenance (with `batch_context` if via `ingestBatch`), rehash, persist, advance Chain 2. |
| `event_id` in store **and** incoming `event_hash` equals stored `provenance[thisHop].arrival_hash` | Emit `ingest.duplicate_received` (§2.9). Do **not** mutate the stored copy. Advance Chain 2 for the duplicate-received event only. |
| `event_id` in store **and** incoming `event_hash` differs from stored `provenance[thisHop].arrival_hash` | Throw `IngestIdentityMismatch`. Abort the batch. |

The identity-match check is O(1): one field comparison between incoming `event_hash` and the stored copy's `metadata.provenance[thisHop].arrival_hash`. The stored `arrival_hash` is by construction the hash this receiver observed on first ingest; if the incoming bytes hash to the same value they are byte-identical to what was first observed, which is sufficient for dedup.

The stored copy is never re-stamped. Its `provenance` chain remains frozen at first-ingest time; its `event_hash` stays stable. Repeated re-receipts accrete `ingest.duplicate_received` events on the destination (each with its own `batch_context` tying it to the re-delivering batch) but leave the subject untouched.

### 2.9 Receiver-originated system events

Two library-originated event types are introduced. Both live under a **receiver-scoped ingest-audit aggregate** — one aggregate per receiver hop, identifier of the form `"ingest-audit:<hop_id>"` (exact tokenization pinned in the implementing REQ). Both are written via the origination path (fresh `event_id`, local `sequence_number`, `provenance[0]` = receiver, Chain 2 fields populated, `batch_context` populated when emitted in response to a batch).

**`"ingest.duplicate_received"`**
Emitted inside `ingestBatch` / `ingestEvent` whenever the idempotency check hits "known + matching" (§2.8). `data`:
```
{
  "subject_event_id": "<uuid>",
  "subject_event_hash_on_record": "<sha256 hex>"
}
```
No payload from the incoming copy is stored — it's already on record, byte-identical by virtue of the `arrival_hash` match. The subject aggregate's materializer does not observe this event (different `event_type`, different `aggregate_id`); the receiver's ingest-audit aggregate is the only place it surfaces. `provenance[0].batch_context` is non-null when emitted via `ingestBatch`; null when emitted via `ingestEvent`.

**`"ingest.batch_rejected"`**
Emitted **only** when the caller invokes `logRejectedBatch` (§2.7). Never emitted implicitly by the library. `data`:
```
{
  "wire_bytes": "<base64>",
  "wire_format": "<format identifier>",
  "byte_length": <int>,
  "wire_bytes_hash": "<sha256 hex>",
  "reason": "<enum: chainBroken | identityMismatch | decodeFailure | backend | ...>",
  "failed_event_id": "<uuid | null>",
  "error_detail": "<freeform string | null>"
}
```
This is the only path by which wire bytes enter the event store — rejected batches that the caller *wants* retained. The happy path produces no wire-bytes duplication. `provenance[0].batch_context` is null (the rejected batch does not produce a populated `BatchContext` — decomposition either failed or was not the outcome that survived).

### 2.10 Failure model

Per approach A from the brainstorm: single-method-per-responsibility, caller composes audit.

**Good path.** `ingestBatch` commits one transaction containing all N subject ingests and any dup-markers. `IngestBatchResult` lists per-subject outcomes. No receiver-originated events other than any `ingest.duplicate_received` emissions.

**Failure path.** `ingestBatch` rolls back and throws. No events from the batch have landed. The caller decides:
- If audit retention is desired: call `logRejectedBatch` with the same bytes + classified reason. A single `ingest.batch_rejected` event commits in an independent transaction, carrying the bytes.
- If audit retention is not desired (debug, test, caller has its own transport-level audit): no further library call. The rejection leaves no event-store record; transport-level logs carry the receipt story.

The library's guarantee is narrower than in the prior design: "if you call `logRejectedBatch`, an auditable record is stored; if you don't, none is." The "every byte accounted for" invariant, if required, is the caller's to maintain by always calling `logRejectedBatch` on failure. For regulated deployments this is a small discipline enforceable by review and integration tests, not a library-internal mechanism.

Within-batch duplicates are **not** failures — each duplicate is a successful ingest outcome that emits a `duplicate_received` audit event and continues processing.

### 2.11 Verification APIs

Two operations, non-throwing, returning a verdict object. Per approach (A) from Question 5:

```dart
/// Walks [event].metadata.provenance from the current tail backward to
/// provenance[0], recomputing each hop's hash against the prior state and
/// verifying it equals that hop's `arrival_hash`. Returns a verdict listing
/// any mismatches. Non-throwing — verification outcomes are data, not
/// control flow. Exceptions (malformed event, IO errors) may still be thrown.
Future<ChainVerdict> verifyEventChain(StoredEvent event);

/// Walks this destination's Chain 2 from [fromIngestSeq] (inclusive) through
/// [toIngestSeq] (inclusive; null = tail), verifying each event's
/// provenance[thisHop].previous_ingest_hash equals the prior event's stored
/// `event_hash`. Returns a verdict listing any mismatches.
Future<ChainVerdict> verifyIngestChain({
  int fromIngestSeq = 0,
  int? toIngestSeq,
});

class ChainVerdict {
  final bool ok;
  final List<ChainFailure> failures;
  const ChainVerdict({required this.ok, required this.failures});
}

class ChainFailure {
  final int position;            // hop index for Chain 1; ingest_sequence_number for Chain 2
  final ChainFailureKind kind;   // enum: arrivalHashMismatch, previousIngestHashMismatch, provenanceMissing, ...
  final String expectedHash;
  final String actualHash;
  const ChainFailure({
    required this.position,
    required this.kind,
    required this.expectedHash,
    required this.actualHash,
  });
}
```

A successful verification: `ChainVerdict(ok: true, failures: [])`. A partial failure: `ok: false`, `failures` populated with every broken link encountered during the walk.

### 2.12 Storage-key implication

On a multi-originator destination, the originator-stamped `sequence_number` is no longer unique — device A's event #1247 and device B's event #1247 both arrive at the same destination. The sembast event-log store's record key cannot be `sequence_number` on a destination; it must be `ingest_sequence_number` (monotone per destination) or an equivalent destination-local identifier.

On an originating mobile device (single originator), `sequence_number` and the local append counter coincide — the existing sembast implementation is unaffected. Only a destination-role `StorageBackend` needs to key by `ingest_sequence_number`. The library carries both counters; which one is used as the storage key is a backend-implementation detail surfaced by the REQ as a requirement on multi-originator destinations.

Per-aggregate fold ordering is unaffected: within a single aggregate, all events share one originator (a single aggregate is edited from one device; multi-source aggregate editing is deferred per memory item 10), so `sequence_number` is monotone within an aggregate on any store. Materializer fold by `sequence_number` per aggregate remains correct on both origins and destinations.

### 2.13 `previous_event_hash` on ingested events

The originator-stamped `previous_event_hash` field (REQ-d00120-B identity field) points to the previous event in the **originator's** log. On a destination that holds events from multiple originators, this field's value does not correspond to any event in the destination's local log — it points to a record in the originator's store, which may or may not be accessible at the destination.

The field is preserved verbatim (identity preservation per §2.4) but is **not** the basis of Chain 2 on destinations. Chain 2 uses `previous_ingest_hash` in receiver provenance entries specifically because the originator-chain field doesn't answer the destination-local question Chain 2 is built for. On a mobile originator (single originator), `previous_event_hash` is the originator's local hash chain and also happens to be the local store's chain — no divergence. Destinations have two conceptually distinct chain concepts coexisting in the record; the library code treats them as orthogonal.

## 3. Caller API summary

```dart
// Wire-side ingest — decomposes, verifies, stamps, persists; throws on failure.
Future<IngestBatchResult> ingestBatch(
  Uint8List bytes, {
  required String wireFormat,
});

// Process-local ingest — no batch_context; throws on failure.
Future<PerEventIngestOutcome> ingestEvent(StoredEvent incoming);

// Caller-composed rejection audit — emits one ingest.batch_rejected event
// carrying the supplied bytes verbatim. Not called by the library itself.
Future<void> logRejectedBatch(
  Uint8List bytes, {
  required String wireFormat,
  required String reason,
  String? failedEventId,
  String? errorDetail,
});

// Auditor / self-check.
Future<ChainVerdict> verifyEventChain(StoredEvent event);
Future<ChainVerdict> verifyIngestChain({int fromIngestSeq = 0, int? toIngestSeq});

class IngestBatchResult {
  final String batchId;   // from the decoded envelope
  final List<PerEventIngestOutcome> events;
}

class PerEventIngestOutcome {
  final String eventId;
  final IngestOutcome outcome;   // enum: ingested | duplicate
  final String resultHash;       // stored event_hash after processing
}

class BatchContext {
  final String batchId;
  final int batchPosition;
  final int batchSize;
  final String batchWireBytesHash;
  final String batchWireFormat;
}

// Typed errors.
class IngestChainBroken implements Exception { ... }
class IngestIdentityMismatch implements Exception { ... }
class IngestDecodeFailure implements Exception { ... }
```

## 4. REQ impact

- **REQ-d00115 (ProvenanceEntry Schema)** — extended with four new assertions specifying `arrival_hash`, `previous_ingest_hash`, `ingest_sequence_number`, `batch_context`: nullability rules, origin-vs-receiver distinction, monotonicity of the ingest counter, immutability (receiver stamps once on ingest, never mutates), and the relationship between `batch_context` and the `esd/batch@1` wire format. Rationale paragraph extended to cover the two-chain framing and the batch-context composition.
- **REQ-d00120 (Canonical Hashing)** — extended rationale covering hash recompute on ingest; existing assertions unchanged in wording (the identity field list in assertion B already includes `metadata`, so appending a provenance entry naturally flows through to a new `event_hash`). A new assertion makes this explicit: "When a receiver appends a `ProvenanceEntry` to `metadata.provenance` during ingest, the event's `event_hash` SHALL be recomputed over the new identity field set and stored. The originator's `event_hash` remains recoverable via the Chain 1 walk specified in REQ-d00145-F."
- **REQ-d00145 (new) — `EventStore.ingestBatch` / `ingestEvent` / `logRejectedBatch` contracts.** Assertions covering: canonical batch envelope format (`esd/batch@1`), per-event verify-first flow, idempotency-by-event-id three-outcome model, emission of `ingest.duplicate_received` inside ingest methods, emission of `ingest.batch_rejected` **only** via `logRejectedBatch`, receiver-scoped ingest-audit aggregate, all-or-nothing transactional semantics within `ingestBatch`, preservation of originator identity fields, Chain 2 stamping on each successful ingest, BatchContext stamping when via `ingestBatch`.
- **REQ-d00146 (new) — `EventStore.verifyEventChain` / `verifyIngestChain` contracts.** Assertions covering: Chain 1 walk semantics (backward from current tail, bottoming out at origin), Chain 2 walk semantics (by `ingest_sequence_number`), `ChainVerdict` shape, non-throwing verdict model, enumeration of failures rather than first-failure abort.

Two new REQ numbers claimed. Current ceiling at spec time is REQ-d00144; Phase 4.9 therefore claims **REQ-d00145** and **REQ-d00146**. Phase 4.11 shifts from its drafted 00146/00147 to 00147/00148 (companion update in `2026-04-24-phase4.11-read-side-api-gaps-design.md`). Phase 4.10 extends REQ-d00128 without claiming a new number. Phase 4.13 (new — see memory) is expected to extend REQ-d00119-B without a new number.

New helper for emitting `ingest.duplicate_received` and `ingest.batch_rejected` events via the existing `_appendInTxn` pattern. Hash-recompute helper that closes over `identity + updated provenance`. ## 7. Risks

### Risk 1: Chain 1 walk complexity
`arrival_hash` verification requires rehashing `identity + provenance[0..K-1]` at each hop during the walk. For a deeply-forwarded event (many hops), the walk is O(N × hash_cost). In practice hop counts will be small (≤ 4 for the foreseeable portal architecture), but the algorithm is worth benchmarking before the cutover phase. Mitigation: test harness includes a 10-hop synthetic case to confirm correctness and a timing check (informational, not a hard bound).

### Risk 2: sembast storage-key migration for destinations
Existing sembast backend behavior keys events by `sequence_number`. Switching destinations to `ingest_sequence_number` is a backend change that mobile does not exercise in production (mobile is originator-only). The risk is the destination-role path diverging from the origin-role path in subtle ways. Mitigation: `ingestEvent` / `ingestBatch` use dedicated storage methods that do not share path with `appendEvent`; the origin-role code is untouched by 4.9.

### Risk 3: `previous_event_hash` on ingested events is a semantic cliff
The field travels verbatim but means something different on a destination than on the originator. A naive reader of destination code might assume it chains the destination's local log and be surprised when it doesn't. Mitigation: the REQ-d00120 rationale is updated to call this out explicitly; any code that touches `previous_event_hash` on a destination carries a comment referencing the REQ text. Chain 2 verification uses `previous_ingest_hash`; `previous_event_hash` has no role in Chain 2 validation.

### Risk 4: Caller forgets to invoke `logRejectedBatch`
Approach A places audit-retention responsibility on the caller. A wire-layer implementation that catches `ingestBatch` exceptions without calling `logRejectedBatch` loses library-level forensic records for rejected batches. Mitigation: docs call out the expected pattern; integration tests exercise the full caller-composed flow; for FDA-regulated deployments the wire layer is code that gets reviewed, and transport-level logs provide a redundant audit path.

### Risk 5: Duplicate-received spam under aggressive retry
An upstream with misconfigured retry could re-send the same batch hundreds of times in a short window, producing hundreds of `ingest.duplicate_received` events (one per subject per retry). The destination's log bloats with dedup noise. Mitigation: a future refinement could add batch-level dedup (short-circuit a re-sent identical batch via `batchWireBytesHash` lookup) before per-subject processing. Deferred; Phase 4.9 accepts per-subject dedup-event emission as the baseline.

### Risk 6: `ingestEvent` bypasses batch-level auditing
A wire-layer caller mistakenly calling `ingestEvent` instead of `ingestBatch` loses the batch-level audit trail (no `batch_context` on the stamped provenance; no way to group events by batch). Mitigation: documentation. The `ingestEvent` docstring specifies in-process-only usage; linter-style checks are out of scope for 4.9.

### Risk 7: Reconstruction of wire bytes depends on canonicalization purity
The audit story for happy-path batches depends on "stored events re-serialize to bytes that hash-match `batchWireBytesHash`." A bug in canonicalization (event JSON, batch envelope, or both) breaks this guarantee. Mitigation: `test/ingest/ingest_batch_reconstruction_test.dart` explicitly verifies round-trip hash equality on a representative batch; a reconstruction test runs against every ingest test to catch drift.

## 8. Out of scope

- Portal HTTP transport, wire-format negotiation, TLS, auth. Portal-side concern; not library.
- Pluggable `BatchCodec` — Phase 4.9 supports exactly one canonical format (`esd/batch@1`). Sponsor-specific wire formats convert to this at the transport layer or await a later codec-registration phase.
- Destination-side materializer / fold strategy for multi-originator event logs. A destination that merely ingests and verifies is complete under Phase 4.9; projecting to a materialized view is a separate design.
- Retention and compaction of ingest-audit events (`ingest.duplicate_received`, `ingest.batch_rejected`). Operationally significant at long-running-destination scale; deferred.
- Cross-originator aggregate-id collision handling. Bundled with multi-source editing (memory item 10).
- Batch-level dedup (short-circuiting a re-sent identical batch without per-subject processing). Refinement deferred.
- Outgoing-side native-format `wire_payload` optimization. REQ-d00119-B currently makes `wire_payload` unconditionally required; relaxing it for FIFO rows whose `wire_format` is the library's native `esd/batch@1` (bytes reconstructible from stored events) is the symmetric dual of this phase's work and belongs in its own phase — see Phase 4.13 in memory.
- `previous_event_hash` field semantics on destinations beyond "preserved verbatim; not part of Chain 2." A destination that wanted to construct its own per-originator linear chain would need additional machinery; not a Phase 4.9 requirement.
- Changes to existing origin-role behavior. `EventStore.append` is unchanged; `StorageBackend.appendEvent` is unchanged. Phase 4.9 is purely additive to the write-path surface.
