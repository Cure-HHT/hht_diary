# Reactive Read Layer (Phase 4.12 Design Spec)

**Date**: 2026-04-25
**Ticket**: CUR-1154 (continuation of the `mobile-event-sourcing-refactor` branch)
**Phase**: 4.12 — follows Phase 4.11 (read-side API gaps). Library + example-app scope.
**Scope**: `event_sourcing_datastore` library — two new abstract methods on `StorageBackend` (`watchEvents`, `watchFifo`) with a library-owned broadcast layer in `SembastBackend`. Migrates three example-app panels off `Timer.periodic` polling.

**Predecessor**: `2026-04-24-phase4.12-reactive-read-layer-stub.md` (the stub flagged five open decisions; this spec resolves them per the decisions log §4.12.A–E).

## 1. Problem (recap from stub)

`event_sourcing_datastore` exposes `Future<List<…>>` reads only. The example app's three read-side panels poll on `Timer.periodic(500ms)`:

- `detail_panel.dart:59,124` — `findAllEvents(limit: 100000)` per tick.
- `event_stream_panel.dart:31` — `findAllEvents(limit: 500)` per tick + sort descending.
- `fifo_panel.dart` — after Phase 4.11 uses `listFifoEntries` + `findEventById`, but still polls because nothing tells it "the queue changed."

Production callers (mobile diary post-cutover, future portal) inherit the same forced-polling pattern. The library has no reactive primitive consumers can subscribe to, no shared coordination point across consumers, and no signal that a write occurred.

## 2. Design

### 2.1 Two reactive APIs, both `Stream`-shaped

Two new abstract methods on `StorageBackend`:

```dart
/// Stream of events appended to the log. Emits each event in
/// `sequence_number` order. On subscribe: first emits replay of every
/// event with `sequenceNumber > afterSequence` (or every event in the
/// log when `afterSequence` is null), then transitions to live emission
/// of events appended or ingested while the subscription is open. The
/// transition is monotone — replayed events are strictly less than the
/// first live event.
///
/// Multiple subscribers SHALL receive the same event sequence (broadcast
/// semantics). The stream remains open until [close] is called on the
/// backend; on backend close all open subscriptions receive `done`.
Stream<StoredEvent> watchEvents({int? afterSequence});

/// Stream of FIFO snapshots for [destinationId]. Emits a snapshot
/// (`List<FifoEntry>` ordered by `sequence_in_queue` ascending) on
/// subscribe and on every mutation to that destination's FIFO
/// (`enqueue`, `appendAttempt`, `markFinal`, `setFinalStatus`,
/// `deleteNullRowsAfterSequenceInQueue`, `deleteFifoStore`). When
/// [destinationId] has no registered FIFO store, the first snapshot is
/// an empty list; subsequent enqueues populate it.
Stream<List<FifoEntry>> watchFifo(String destinationId);
```

Both are `Stream`, not `ValueListenable` or `ChangeNotifier` — pure-Dart, no Flutter dependency, broadcast-friendly, and consumers in the Flutter widget tree can wrap with `StreamBuilder` cheaply.

### 2.2 Library-owned broadcast (not sembast onSnapshots)

`SembastBackend` owns two `StreamController.broadcast()` instances:

- `_eventsController: StreamController<StoredEvent>` — fed by every successful `appendEvent` and `appendIngestedEvent` after commit.
- `_fifoChangesController: StreamController<String>` — fed by every successful FIFO mutation (the destinationId is the payload). `watchFifo(destId)` filters this stream and re-fetches `listFifoEntries(destId)` on each matching emission.

Why library-owned, not sembast-native:

1. **Backend-agnostic contract.** Future `PostgresBackend` implements the same `Stream` shape via its own write paths (or `LISTEN/NOTIFY`); abstract `StorageBackend` doesn't leak sembast types.
2. **Coupling avoidance.** Sembast's `onSnapshots` returns a sembast-typed cursor; using it in the public contract would force every alternative backend to translate to sembast's shape.
3. **Outside-write risk is bounded.** Library code is the only sanctioned writer; raw test-only writes via `debugDatabase()` bypass the broadcast (acceptable — tests own their own assertions). The `queryAudit` reader code does NOT write, so it doesn't create a notification gap.

### 2.3 Replay semantics — `watchEvents`

On subscribe with `afterSequence: X`:

1. Open a transformer holding the last replayed sequence number (initially `X`, or `0` if `null`).
2. Synchronously begin replay: read `findAllEvents(afterSequence: X)`, emit each in order, advancing the held sequence.
3. As soon as the replay completes, attach a listener to `_eventsController.stream`. Filter live events to `event.sequenceNumber > heldSequence` (covers the race where new events appended during replay would otherwise be missed).
4. Each live event passing the filter is emitted; held sequence advances.

This is "tail -F" semantics: replay-then-live, monotonically ordered, no duplicates at the boundary. Standard pattern; widely understood. The lib does NOT retain per-subscriber state across pause/resume — slow consumers that pause may miss events; they recover by canceling and resubscribing with their last-seen sequence.

### 2.4 Snapshot-on-subscribe — `watchFifo`

On subscribe:

1. Synchronously emit the current snapshot via `listFifoEntries(destinationId)`.
2. Filter `_fifoChangesController.stream` to events whose payload equals `destinationId`. On each match, re-fetch `listFifoEntries(destinationId)` and emit.

Snapshot-then-changes semantics, like `LiveData` / `StateFlow.value`. Consumers that want only-deltas can compare against the prior emission; the lib doesn't expose a per-row delta primitive.

Debouncing is NOT implemented in the library — emissions fire 1:1 with FIFO mutations. The example panels can wrap with their own debouncer if rendering cost matters; at mobile scale (FIFO mutations happen at user-action rate or drain-tick rate, neither of which is high-frequency) raw 1:1 emission is acceptable.

### 2.5 Coordination — shared backend instance, no new primitive

The library does NOT add a separate "DataStore" handle to coordinate reads across consumers. The existing `SembastBackend` instance IS the coordination point: a Flutter app constructs one backend, hands it to N panels, and broadcast streams naturally deduplicate the change-detection work (one underlying controller, N subscribers).

Document in the new APIs' dartdoc: **"Consumers SHALL share a single `StorageBackend` instance to benefit from broadcast deduplication; constructing multiple backends over the same database file is unsupported."**

### 2.6 Error model

Storage errors observed during a snapshot fetch (e.g., `watchFifo` re-fetches `listFifoEntries` and the read fails for a non-fatal reason) propagate via `Stream.addError` — the stream stays open, downstream `StreamBuilder` shows the error, recovery is the caller's call. Fatal errors that mean the backend is unusable (e.g., `close()` was called or the underlying database is gone) cause `_eventsController.close()` and `_fifoChangesController.close()`, which sends `done` to all subscribers.

The decisions log (§4.12 deferred items) records this; assertion text in REQ-d00149-D / REQ-d00150-D codifies it.

### 2.7 Lifecycle integration with `SembastBackend.close`

`SembastBackend.close()` currently awaits the underlying database close. Phase 4.12 adds:

- Before closing the database: call `await _eventsController.close()` and `await _fifoChangesController.close()`. This drains in-flight emissions and signals `done` to subscribers.
- The order matters: close the controllers first so that an in-progress `appendEvent` doesn't try to add to a closed controller after `_db.close()` returns. Concretely: close happens after `_db.close()` returns, but the controller close runs before — the close method becomes:

  ```dart
  Future<void> close() async {
    await _eventsController.close();
    await _fifoChangesController.close();
    await _db.close();
  }
  ```

  Calls to `watchEvents` / `watchFifo` after `close()` SHALL throw `StateError('SembastBackend.close has been called')` — caller bug; fail fast.

### Tests

Library tests:

- `test/storage/sembast_backend_watch_events_test.dart` (new) — covers REQ-d00149 assertions:
  - Replay-then-live: pre-existing events are emitted in order on subscribe, then a live append produces the next emission.
  - `afterSequence` filter: pre-existing events with `seq <= afterSequence` are NOT replayed.
  - Multiple subscribers receive identical sequences (broadcast).
  - Ingested events (via `appendIngestedEvent`) are emitted to subscribers.
  - `close()` sends `done` to active subscribers.
  - Subscribing after `close()` throws.
- `test/storage/sembast_backend_watch_fifo_test.dart` (new) — covers REQ-d00150 assertions:
  - Snapshot on subscribe (empty for unknown destination; populated for known).
  - Each FIFO mutation produces a new snapshot emission (enqueue / markFinal / appendAttempt / tombstone-trail-delete).
  - Multiple subscribers receive identical sequences.
  - Cross-destination isolation (mutating destination A does not emit to a `watchFifo(B)` subscriber).
  - `close()` sends `done`.

## 5. Out of scope

- **`watchEntry(aggregateId)`** (per-aggregate materialized view stream) — deferred to mobile cutover (CUR-1169) per decisions log §4.12.D.
- **Sembast `onSnapshots` integration** — explicitly chose library-owned broadcast.
- **Coordination primitive separate from `StorageBackend`** — explicitly rejected.
- **Compaction / log-truncation semantics** — not on roadmap.
- **Cross-device subscription** — orthogonal to local reactive reads.
- **Per-row FIFO delta primitive** — `watchFifo` emits whole snapshots; consumers that want deltas diff against the prior emission.
- **Debouncing in the library** — emissions are 1:1 with mutations; consumers debounce if they need to.
- **Backpressure beyond what `StreamController.broadcast` provides** — no buffering; slow consumers may lose events between pause/resume and recover via re-subscription with `afterSequence`.

## 6. Risks

### Risk 1: Race between transaction commit and broadcast emission

If `_eventsController.add(event)` runs after the storage transaction returns but before the broadcast is delivered, a fast subscriber doing a `findAllEvents` read inside the listener might see the event already in the store. This is correct behavior, not a race — emit-after-commit guarantees observability. Mitigation: the implementation MUST place `controller.add` AFTER the `await backend.transaction(...)` returns, never inside the transaction body. Code-review checklist item.

### Risk 2: Listener-thrown exceptions tearing down the controller

A subscriber whose `onData` throws would, by default Stream semantics, NOT tear down the broadcast — `StreamController.broadcast()` isolates listener errors. But this depends on using `broadcast()` not the single-subscriber controller. Code-review checklist item: the controllers MUST be `StreamController.broadcast()`.

### Risk 3: `watchEvents` race during replay

If an event is appended between the `findAllEvents(afterSequence: X)` read and the controller-stream subscription, it could be missed (replay didn't see it; live-stream filter at `seq > lastReplayed` would catch it). The held-sequence filter in §2.3 covers this; the test suite SHOULD include a "concurrent append during subscribe" test.

### Risk 4: `watchFifo` snapshot cost

Each FIFO mutation triggers a `listFifoEntries(destId)` call per subscriber. At N subscribers per destination, that's N reads per mutation. Mitigation 1: at mobile scale this is fine (one panel, one mutation rate). Mitigation 2 (NOT in 4.12): a future shared-cache layer could deduplicate the read, falling under the deferred coordination primitive.

### Risk 5: Test flakiness

Stream-based tests are easier to flake than `Future`-based ones. Mitigation: use `expectLater(stream, emitsInOrder([...]))` from `package:flutter_test`; avoid `Future.delayed` waits; use `pumpEventQueue` only when necessary; close subscriptions in `tearDown`. Document the pattern in the new test files.
