# Outgoing Native-Format `wire_payload` Optimization (Phase 4.13 Design Spec)

**Date**: 2026-04-25
**Ticket**: CUR-1154 (continuation of the `mobile-event-sourcing-refactor` branch)
**Phase**: 4.13 — final phase before mobile cutover (CUR-1169). Library-only.
**Scope**: `event_sourcing_datastore` library — storage-side optimization that drops `wire_payload` for native `esd/batch@1` rows, persists envelope metadata instead, and reconstructs bytes at drain time via `findEventById` + `BatchEnvelope.encode`. Mobile-only. Symmetric dual of Phase 4.9's ingest-side canonical batch format.

**Predecessor**: `2026-04-24-phase4.13-outgoing-native-wire-payload-stub.md` (the stub flagged choices around envelope-metadata storage and drain branching; this spec resolves them per decisions log §4.13.A–F).

**Depends on**: Phase 4.9 (`BatchEnvelope.decode`/`encode` exist) and Phase 4.11 (`findEventById` exists).

## 1. Problem (recap from stub)

REQ-d00119-B mandates that every FIFO entry carry a non-null `wire_payload`. The field exists for retry determinism, lossy 3rd-party formats, and send-log integrity. Phase 4.9 introduced `esd/batch@1` — a canonical batch format that IS round-trippable: same events + same envelope metadata + JCS canonicalization (RFC 8785) yields byte-identical wire bytes on every encoding pass.

For FIFO rows whose `wire_format == "esd/batch@1"`, storing `wire_payload` duplicates information that is losslessly reconstructible from the row's `event_ids` + a small envelope metadata block. On mobile, the highest-volume destination is the diary server (native), so most FIFO rows pay this duplication tax.

Concrete savings estimate (from stub §3): 100 native rows/day × 4.8KB/row = ~500KB/day per device. Compounds with retention.

## 2. Design

### 2.1 Storage-side optimization, transparent to `Destination` API

The destination's `transform(batch) → WirePayload` continues to return canonical bytes with `contentType: 'esd/batch@1'` — no destination contract change. The library's `enqueueFifoTxn` does the optimization opportunistically:

- If `wirePayload.contentType == 'esd/batch@1'`: parse the bytes via `BatchEnvelope.decode`, extract envelope metadata into a typed `BatchEnvelopeMetadata` value object, persist the metadata + drop the raw bytes (`wire_payload = null`).
- Otherwise: store `wire_payload` as-is, `envelope_metadata = null`.

Drain reverses the transformation:

- If `fifoEntry.envelopeMetadata != null`: look up each event in `event_ids` via `findEventById`, build a `BatchEnvelope` with the stored metadata + recovered events (as `StoredEvent.toMap()`), call `.encode()` to produce wire bytes. Send.
- Else: send `fifoEntry.wirePayload` as-is.

3rd-party destinations (sponsor CSV, Rave EDC XML, etc.) that produce non-`esd/batch@1` bytes are completely untouched by this change.

### 2.2 New `BatchEnvelopeMetadata` value object

Carries everything from `BatchEnvelope` except `events`:

```dart
class BatchEnvelopeMetadata {
  const BatchEnvelopeMetadata({
    required this.batchFormatVersion,
    required this.batchId,
    required this.senderHop,
    required this.senderIdentifier,
    required this.senderSoftwareVersion,
    required this.sentAt,
  });

  final String batchFormatVersion;  // always "1" for esd/batch@1
  final String batchId;
  final String senderHop;
  final String senderIdentifier;
  final String senderSoftwareVersion;
  final DateTime sentAt;

  /// Build from a parsed BatchEnvelope (drops events).
  factory BatchEnvelopeMetadata.fromEnvelope(BatchEnvelope env);

  /// Reconstruct a full BatchEnvelope by attaching events (drain path).
  BatchEnvelope toEnvelope(List<Map<String, Object?>> events);

  Map<String, Object?> toMap();
  factory BatchEnvelopeMetadata.fromMap(Map<String, Object?> m);
}
```

Lives at `lib/src/destinations/batch_envelope_metadata.dart` (alongside `WirePayload`) so it ships in the destinations package alongside the FIFO contract that uses it.

### 2.3 `FifoEntry` schema additions

Add one optional field:

```dart
final BatchEnvelopeMetadata? envelopeMetadata;
```

Persisted as a sub-map under the `envelope_metadata` key in the sembast row. JSON-serializable via the value object's `toMap`/`fromMap`.

The existing `wirePayload` field becomes nullable (already typed `WirePayload` — reuse the existing type but allow null at the top level). Same treatment in `FifoEntry.fromJson` / `toJson`.

### 2.4 Retry determinism

Per decisions log §4.13.D: re-encoding on every drain attempt is byte-deterministic because:

1. `BatchEnvelopeMetadata` is stored at enqueue and reused verbatim across retries (same `batch_id`, same `sent_at`, same sender identity).
2. Events are immutable (REQ-p00004); `findEventById(eventId)` returns the same `StoredEvent` on every call.
3. `StoredEvent.toMap()` is deterministic (same input → same output).
4. JCS canonicalization (RFC 8785) is byte-deterministic by construction.

A retry sees identical bytes to the original send. Idempotency on the destination is preserved.

### 2.5 Drain implementation

The drain loop (`lib/src/sync/drain.dart`) reads the head FIFO entry and currently calls `destination.send(entry.wirePayload)`. Phase 4.13 inserts a wire-bytes resolution step:

```dart
final WirePayload payload;
final envelope = head.envelopeMetadata;
if (envelope != null) {
  // Native rebuild from event_ids + stored envelope.
  final events = <Map<String, Object?>>[];
  for (final eventId in head.eventIds) {
    final ev = await backend.findEventById(eventId);
    if (ev == null) {
      throw StateError(
        'native FIFO row references missing event $eventId',
      );
    }
    events.add(ev.toMap());
  }
  final bytes = envelope.toEnvelope(events).encode();
  payload = WirePayload(
    bytes: bytes,
    contentType: BatchEnvelope.wireFormat,
    transformVersion: head.transformVersion,
  );
} else {
  // 3rd-party path: bytes were stored verbatim.
  payload = head.wirePayload!;
}
final result = await destination.send(payload);
```

The "missing event" `StateError` is a hard-fail diagnostic — referenced events exist in the log because the FIFO row was built from them; absence indicates a corruption or out-of-band deletion. The orchestrator can surface this; we don't add a second recovery path.

### 2.6 Spec change — REQ-d00119-B rewrite + REQ-d00119-K addition

REQ-d00119-B today reads:

> B. A FIFO entry SHALL carry the fields `entry_id`, `event_ids`, `event_id_range`, `sequence_in_queue`, `wire_payload`, `wire_format`, `transform_version`, `enqueued_at`, `attempts[]`, `final_status`, and `sent_at`. The `event_ids` and `event_id_range` fields hold the batch contract defined in REQ-d00128; a single-event batch is a batch of length one.

Rewritten in final-state voice (no transition language):

> B. A FIFO entry SHALL carry the fields `entry_id`, `event_ids`, `event_id_range`, `sequence_in_queue`, `wire_payload`, `wire_format`, `transform_version`, `enqueued_at`, `attempts[]`, `final_status`, `sent_at`, and `envelope_metadata`. The `event_ids` and `event_id_range` fields hold the batch contract defined in REQ-d00128; a single-event batch is a batch of length one. `wire_payload` SHALL be non-null when `wire_format != "esd/batch@1"` (3rd-party destinations whose serialization is not round-trippable); `wire_payload` SHALL be null when `wire_format == "esd/batch@1"` (native destinations whose payload is reconstructible from `event_ids` and `envelope_metadata`).

New assertion REQ-d00119-K:

> K. `envelope_metadata` SHALL be a `BatchEnvelopeMetadata` value carrying `batch_format_version`, `batch_id`, `sender_hop`, `sender_identifier`, `sender_software_version`, `sent_at`. It SHALL be non-null when `wire_format == "esd/batch@1"` and null otherwise. The values SHALL be set at enqueue time and SHALL NOT be mutated thereafter — they are part of the FIFO row's identity for retry determinism. Drain reconstructs the wire bytes by combining `envelope_metadata` with `event_ids`-resolved events through `BatchEnvelope.encode`; the encoding is deterministic across retries (RFC 8785 JCS).

## 6. Risks

### Risk 1: Encoding non-determinism

Mitigation: end-to-end test that enqueues a native row, drains twice with mocked send capturing bytes, asserts bytes are byte-equal. Catches any latent non-determinism in `StoredEvent.toMap`, JCS canonicalization, or envelope construction.

### Risk 2: Missing event at drain time

Native row references `event_ids: [a, b, c]`; one of them returns null from `findEventById` (out-of-band deletion, corruption). Per §2.5, drain throws `StateError`. The test suite includes this case.

### Risk 3: Schema-shape backward compatibility for existing tests

Test fixtures that build FifoEntry with all-positional or all-named arguments may break when a new optional field is added. Mitigation: greenfield + the new field is OPTIONAL in the constructor (default null). Tests that don't care about the field don't need to change.

### Risk 4: `BatchEnvelope.decode` overhead at enqueue time

Decode parses the JSON and validates. It runs once per enqueue (not on hot drain path). For batches of typical size (a few KB), parse time is microseconds. No measurable impact.
