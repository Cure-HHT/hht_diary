# Phase 4.22 Design: Materialize-on-Ingest and System-Event Bridging

**Date:** 2026-04-26
**Ticket:** CUR-1154 (closing work)
**Status:** Approved (brainstorm 2026-04-26)
**Depends on:** Phase 4.9 (sync-through ingest), Phase 4.13 (outgoing native wire payload), Phase 4.14 (arch cleanup), Phase 4.16 (event versioning), Phase 4.17 (config-change audit events), Phase 4.18 (syncpolicy hot-swap), portal-pane (dual-pane example).

## Intent

Phase 4.9 deferred materialization on the ingest path: a destination that ingests events writes them to its event log but does not project them through any registered `Materializer`, leaving the receiver's read model permanently stale relative to its event log. Phase 4.17 hard-filtered all reserved system audit events from every destination FIFO at fill time, so a destination that wishes to forensically observe an upstream node's local-state mutations (FIFO tombstones, destination registry changes, retention sweeps) cannot do so over the wire.

This phase closes both gaps and adds the discrimination machinery that makes them safe:

- **A. Materialize-on-ingest.** `EventStore.ingestBatch` and `EventStore.ingestEvent` fire the same per-event materializer loop already present in `EventStore.append`. Receivers maintain a materialized view automatically as events arrive — no manual `rebuildView` required.
- **B. System events on the wire.** Destinations may opt in (per-`SubscriptionFilter`) to bridging system audit events alongside user events. The receiver stores them; the receiver does NOT replay them as commands against its own local registries.
- **C. Discrimination + install identity.** `Source.identifier` is documented as the per-installation unique identity (UUIDv4 in production). System events use `source.identifier` as their `aggregate_id` so two installations' system streams never collide on the receiver. New API surface (`StoredEvent.originatorHop` getter, `EventStore.isLocallyOriginated` method, `findAllEvents(originatorHopId:, originatorIdentifier:)` query) lets downstream code distinguish locally-originated events from upstream-bridged events without writing provenance navigation by hand.

The dual-pane demo demonstrates all three end-to-end: a hop badge in the event stream panel, a second mobile-side `Native` destination subscribed to system audits, and the portal pane's `MaterializedPanel` populating automatically as mobile records entries.

## Architecture

```text
       MOBILE PANE                              PORTAL PANE
       (install UUID = M-uuid)                  (install UUID = P-uuid)

  user click -> EventStore.append           DownstreamBridge -> EventStore.ingestBatch
       |                                          |
       | atomic txn:                              | atomic txn:
       |   1. appendEvent                         |   1. appendEvent (per event in batch)
       |   2. for m in materializers:             |   2. for m in materializers:
       |        if def.materialize                |        if def.materialize
       |          and m.appliesTo(event)          |          and m.appliesTo(event)
       |          -> apply (NEW)                  |          -> apply (Phase 4.22 NEW)
       v                                          v
  event_log                                  event_log
  diary_entries (materialized view)          diary_entries (materialized view, NEW)

  fillBatch per destination:
       NativeUser FIFO  ----wire--->  ingestBatch
         (entryTypes: [...user...],
          includeSystemEvents: false)
       NativeAudit FIFO ----wire--->  ingestBatch
         (entryTypes: [],
          includeSystemEvents: true)         <-- aggregate_id = M-uuid for mobile audits
                                                 stored alongside portal's own
                                                 aggregate_id = P-uuid system events
```

Three points of leverage:

1. **`_ingestOneInTxn` runs materializers** with the same gates as `_appendInTxn` (`def.materialize` AND `m.appliesTo(event)`). System entry types ship with `materialize: false`, so system events never project; user events project per the existing materializer contract.

2. **`SubscriptionFilter.includeSystemEvents`** is a new boolean (default `false`) that, when `true`, bypasses the `entryTypes` list for the 10 reserved system entry types (they all pass through). When `false`, the existing Phase 4.17 hard-drop behavior is preserved. The lift is per-destination opt-in, never global.

3. **`source.identifier` IS the system aggregate_id.** All 10 audit emission sites use `source.identifier` as the aggregate_id rather than per-registry strings. One per-installation aggregate carries the install's complete system event chain end-to-end. Across hops, each install's chain is independently rooted at its own UUID; they never collide on the receiver.

## Design

### §1. Materialize-on-ingest

`_ingestOneInTxn` (event_store.dart:602+) gains a materializer loop modelled after `_appendInTxn` (event_store.dart:452-473):

```text
final def = registry.byId(event.entryType);
if (def != null && def.materialize) {
  for (final m in materializers) {
    if (!m.appliesTo(event)) continue;
    final target = await m.targetVersionFor(txn, backend, event.entryType);
    final promoted = m.promoter(
      entryType: event.entryType,
      fromVersion: event.entryTypeVersion,
      toVersion: target,
      data: event.data,
    );
    await m.applyInTxn(
      txn,
      backend,
      event: event,
      promotedData: promoted,
      def: def,
      aggregateHistory: List<StoredEvent>.unmodifiable(aggregateHistory),
    );
  }
}
```

The loop runs per-event inside the existing ingest transaction. Both `ingestBatch` (multi-event wire path) and `ingestEvent` (process-local single-event path) benefit. A materializer or promoter throw rolls back the entire batch (REQ-d00145-A all-or-nothing preserved); REQ-d00140-H ("EntryPromoter exception propagates without wrapping") applies symmetrically.

The materializer is origin-agnostic: `Materializer.apply` is a pure function of `(previous, event, def, firstEventTimestamp)` per REQ-d00121-A. Whether the event arrived via local append or via ingest is invisible to the materializer; both produce the same view-row shape from the same input. A portal that ingests user-aggregate events from multiple mobile installations therefore maintains one unified materialized view, keyed by the user-aggregate UUIDv7s (which are collision-free across installs per existing convention).

Cross-cutting consequence for the demo's portal pane: `MaterializedPanel` populates automatically as mobile records demo_note entries; no `Rebuild view` button click required. The Phase 4.9 design spec §398 ("destination-side materializer / fold strategy ... is a separate design") is closed by this section.

### §2. System events on the wire — opt-in

`SubscriptionFilter` gains one field:

```text
class SubscriptionFilter {
  const SubscriptionFilter({
    this.entryTypes = const <String>[],
    this.includeSystemEvents = false,    // NEW
  });

  final List<String> entryTypes;
  final bool includeSystemEvents;        // NEW

  bool matches(StoredEvent event) {
    if (kReservedSystemEntryTypeIds.contains(event.entryType)) {
      return includeSystemEvents;        // bypass entryTypes filter
    }
    return entryTypes.contains(event.entryType);
  }
}
```

The hard-drop in `fillBatch` (sync/fill_batch.dart:125) and `historicalReplay` (sync/historical_replay.dart:111) is removed. Filter dispatch defers entirely to `SubscriptionFilter.matches`, which now has explicit handling for the system-entry-type case.

Time-window filter (`event.client_timestamp >= dest.startDate`) still applies — a destination registered today does not get historical system events from before its `startDate`. This matches REQ-d00129-I user-event semantics; nothing about system events overrides it.

The current Phase 4.17 default-off behavior is preserved: every existing `SubscriptionFilter` instance has `includeSystemEvents: false` (the default), so no test or caller observes a behavior change unless it explicitly opts in.

#### §2.1. Multi-Native pattern

The lib has no cardinality constraint on `Native` (or any other) destinations. The demo concretely instantiates two on the mobile pane to demonstrate independent FIFOs with independent subscription policies:

```text
NativeDemoDestination(
  id: 'NativeUser',
  filter: SubscriptionFilter(
    entryTypes: <String>['demo_note', 'red_button_pressed', ...],
    includeSystemEvents: false,
  ),
  bridge: portalBridge,
)

NativeDemoDestination(
  id: 'NativeAudit',
  filter: const SubscriptionFilter(
    entryTypes: <String>[],
    includeSystemEvents: true,
  ),
  bridge: portalBridge,
)
```

Both feed the same `DownstreamBridge`; the lib does not know they end at the same receiver. **Wedge isolation use case**: a poison user event that wedges `NativeUser` leaves `NativeAudit` free to keep streaming `system.fifo_tombstoned_and_refilled` audits to portal so an operator dashboard observes the wedge as it happens. This is the architectural justification for not hardcoding "Native always bridges system events" — operators may want to separate user-data delivery from out-of-band signaling, and the lib should not foreclose that.

### §3. Discrimination and install identity

#### §3.1. `Source.identifier` is the install identity (documentation contract)

`Source.identifier` is documented as MUST be globally unique per installation. Recommended shape: UUIDv4 generated on first app install, persisted in app preferences (or platform-equivalent durable storage), passed to `bootstrapAppendOnlyDatastore` on every subsequent boot.

The lib does not validate format — callers that violate the global-uniqueness requirement get correct lib behavior on each install individually but produce data that collides when bridged together. This is a permanent caller obligation, parallel to REQ-d00142-C's treatment of `softwareVersion` ("`Source` SHALL NOT validate this at runtime — the shape is a permanent caller obligation enforced downstream").

The demo's existing `'demo-device'` / `'demo-portal'` strings are removed in favor of per-pane persisted UUIDv4s (see §5).

`hopId` retains its current meaning: well-known role-class string. Two mobile installs both ship `hopId: 'mobile-device'` and are correctly distinguished at the receiver by their differing `identifier` values.

#### §3.2. System aggregate_id convention

The 10 reserved system entry types append events with `aggregate_id = source.identifier`. Audit emission sites updated:

- `bootstrap.dart` — `system.entry_type_registry_initialized` (1 site)
- `destination_registry.dart` — `addDestination`, `setStartDate`, `setEndDate`, `deactivateDestination`, `deleteDestination`, `tombstoneAndRefill` (6 sites)
- `event_store.dart` — `applyRetentionPolicy`, `clearSecurityContext`, security context lifecycle audits (3 sites)

All 10 sites already source the local `Source` via dependency injection; the change is mechanical (read `source.identifier` instead of computing a per-registry constant).

Result: per installation, all system events live in a single hash-chained aggregate. The chain protects every system event for the install's lifetime end-to-end. After bridging, the receiver's event log holds one aggregate per install that has ever bridged to it (each chain intact, none colliding).

Cross-install queries that want "all destination_added events from any install" filter by `entry_type = 'system.destination_registered'` — the entry_type field is stable across installs and is the right discriminator for "what kind of system action."

#### §3.3. dedupeByContent semantic refinement

Current `dedupeByContent` (event_store.dart:380-399) loads `aggregateHistory.last` regardless of entry_type and compares content. Under §3.2, multiple system entry types share an aggregate; the "last" event will frequently be a different entry type than the candidate, so content hashes always differ → dedupe never fires.

The fix is to compare against the most-recent event of **matching entry_type** within the aggregate:

```text
final prior = aggregateHistory
    .reversed
    .firstWhereOrNull((e) => e.entryType == entryType);
if (dedupeByContent && prior != null) {
  // ... existing content-hash compare
}
```

This is the semantic dedupe actually wants; the current "last of any type" was incidentally correct only because of single-purpose aggregates. The behavior contract change matters because:

- Existing callers using `dedupeByContent: true` currently use it only for `system.entry_type_registry_initialized` (single-purpose aggregate). The refined semantic preserves their intent: same entry_type, same aggregate, same content → still skipped.
- Future callers gain the ability to use `dedupeByContent` on shared aggregates without surprise.

#### §3.4. Discrimination API

**Sugar getter on `StoredEvent`** (b):

```text
extension StoredEventOriginExt on StoredEvent {
  /// First provenance entry — the originator's hop.
  ProvenanceEntry get originatorHop {
    final p = provenance;
    if (p.isEmpty) {
      throw StateError('StoredEvent has empty provenance; expected at least originator');
    }
    return p.first;
  }
}
```

REQ-d00115 requires every event to carry at least one `ProvenanceEntry` (the originator's), so the throw is defensive.

**Method on `EventStore`** (c):

```text
/// True iff [event] was originated locally on this EventStore's `Source`.
/// Compares originator install identity (provenance[0].identifier) against
/// the EventStore's bound source.identifier — not hopId, since two installs
/// of the same role-class are distinct originators.
bool isLocallyOriginated(StoredEvent event) =>
    event.originatorHop.identifier == source.identifier;
```

Identity comparison is on `identifier` (the install UUID), not `hopId` (the role class). Two mobile installs are not "the same originator."

**Storage-level query support** (d):

```text
Future<List<StoredEvent>> findAllEvents({
  int? afterSequence,
  String? originatorHopId,        // matches provenance[0].hopId       (class)
  String? originatorIdentifier,   // matches provenance[0].identifier  (install)
});
```

Both nullable; AND'd together if both supplied. Implementation projects `provenance[0].hopId` and `provenance[0].identifier` as filterable fields on the sembast event-row schema, OR uses sembast's JSON-path filtering capability — implementation choice deferred to plan; both work and the API surface is identical either way.

Use cases:
- `findAllEvents(originatorIdentifier: '<install-uuid>')` — forensic "what did this exact install do."
- `findAllEvents(originatorHopId: 'mobile-device')` — dashboard "all mobile-class installs."
- `findAllEvents(originatorIdentifier: source.identifier)` — equivalent to "locally-originated only" as a query.

#### §3.5. Receiver-stays-passive invariant (formalized)

**Bridged audits are informational about the originator's local state; the receiver's local state is unaffected.** Concretely:

- Receiver storing `system.destination_registered` from mobile does NOT call `addDestination` on its local `DestinationRegistry`.
- Receiver storing `system.entry_type_registry_initialized` from mobile does NOT modify its local `EntryTypeRegistry`.
- Receiver storing `system.fifo_tombstoned_and_refilled` from mobile does NOT touch any of its local FIFOs.
- Receiver storing `system.retention_policy_applied` from mobile does NOT trigger a local retention sweep.

This is already true today (registry mutations are exclusively API-driven, not event-driven), but it becomes a formal invariant: **`DestinationRegistry`, `EntryTypeRegistry`, and FIFO state SHALL NOT be mutated by `EventStore.ingestBatch` or `EventStore.ingestEvent` paths.**

This invariant is the architectural reason cross-aggregate stream projection (a future ticket) is the right shape for cross-hop dashboards — projections are read-side derivations from the event log, never commands against local mutable state.

#### §3.6. Hash chain integrity under bridging

Per Phase 4.9, ingested events keep the originator's `previous_event_hash` verbatim (it points to the originator's local prior event for that aggregate). Receiver recomputes `event_hash` (chain 2). Under §3.2, mobile's system events form a chain rooted at mobile's `source.identifier`; portal's own system events form a separate chain rooted at portal's `source.identifier`. Bridging copies mobile's chain to portal verbatim under mobile's aggregate_id; chain 1 verification (per-event provenance) and the per-aggregate `previous_event_hash` walk both remain valid on the receiver because we preserve the originator's chain ordering and aggregate identity end-to-end.

No new hash-chain mechanism is needed; the existing Phase 4.9 properties extend naturally to system events under the new aggregate-id convention.

### §4. Demo updates

#### §4.1. Per-pane install UUID

`main.dart` removes hardcoded `'demo-device'` / `'demo-portal'` strings. Each pane independently reads-or-mints a UUIDv4 stored at `<appSupportDir>/event_sourcing_datastore_demo/<paneLabel>.install.uuid` (plain text, one UUID per file). On first boot the file does not exist; the demo calls `Uuid().v4()` and writes the result. On subsequent boots the file is read back. The two panes' UUIDs are independent — neither derives from nor shares any byte with the other.

```text
Future<String> _readOrMintUUID(String path) async {
  final f = File(path);
  if (await f.exists()) return (await f.readAsString()).trim();
  final id = const Uuid().v4();
  await f.writeAsString(id);
  return id;
}

final mobileInstallUUID = await _readOrMintUUID(
  p.join(demoDir.path, 'MOBILE.install.uuid'),
);
final portalInstallUUID = await _readOrMintUUID(
  p.join(demoDir.path, 'PORTAL.install.uuid'),

final mobile = await _bootstrapPane(

final portal = await _bootstrapPane(

    hopId: 'portal-server',
    identifier: portalInstallUUID,
    softwareVersion: 'event_sourcing_datastore_demo@0.1.0+1',
  ),
  ...
);
```

1. Boot demo. Expect: 7 system events on each pane (existing behavior, just under new `aggregate_id` = pane's install UUID).

### §5. Out of scope

- **Cross-aggregate stream projection** (entry-type-keyed dashboards aggregating across hops, e.g., "how often are mobile apps tombstoning their FIFOs?"). The existing `Materializer` abstraction consolidates per-`aggregate_id`, which is the wrong shape for this use case. Deferred to a future ticket; the ingredient pieces (`findAllEvents(originatorHopId:)` + `findAllEvents(originatorIdentifier:)` + `entry_type` filter) are sufficient for ad-hoc Dart-side aggregation in the meantime.
- **Replication of registry mutations across hops** (the receiver storing a `destination_registered` audit does NOT call `addDestination` locally). §3.5 formalizes this as a permanent invariant.
- **Wire-format evolution / negotiation for system event payloads.** Handled by the still-stub Phase 4.20 design.
- **Bidirectional sync** (portal-pane Natives bridging back to mobile). Out of scope per existing project memory.

### Existing REQs extended (assertion text edits or additions)

**REQ-d00128 — FIFO Batch Shape and Fill Cursor**

**REQ-d00142 — Source Stamping Provenance Identity**
- Add assertion (expected D): `Source.identifier` SHALL be the per-installation unique identity. **REQ-d00145 — EventStore Ingest Contract**
- Add assertion (expected N): `EventStore.ingestBatch` and `EventStore.ingestEvent` SHALL fire materializers per-event inside the existing ingest transaction, with the same gates as `EventStore.append` (`def.materialize` flag and `m.appliesTo(event)` predicate). A materializer or promoter throw SHALL cause the entire batch transaction to roll back per REQ-d00145-A. (Cross-references the new REQ-d00121-K assertion.)

### New REQ to claim (REQ-dXXXXX — Cross-Hop Event Discrimination and Bridging)

Title: **Cross-Hop Event Discrimination and Bridged System-Event Storage**

Rationale: Phase 4.22 introduces the API surface and storage-level conventions that let a receiver distinguish locally-originated events from bridged-from-upstream events without writing provenance navigation by hand, and that make the dashboard / forensic query patterns ("all events from this install", "all events from any install of this hop class") efficient.

Proposed assertions:

- **A.** `StoredEvent.originatorHop` SHALL return the first `ProvenanceEntry` in the event's provenance chain (the originator's hop). Implementations SHALL throw `StateError` if `provenance` is empty.
- **B.** `EventStore.isLocallyOriginated(StoredEvent event)` SHALL return `true` iff `event.originatorHop.identifier == source.identifier` (the install UUID of the `EventStore`'s bound `Source`). The comparison SHALL be on `identifier`, not `hopId` — two installations of the same role class are distinct originators.
- **C.** `StorageBackend.findAllEvents` SHALL accept two new optional named parameters: `String? originatorHopId` (matches `provenance[0].hopId`) and `String? originatorIdentifier` (matches `provenance[0].identifier`). When both are supplied, results SHALL match both (AND semantics). Implementations MAY project the two fields onto queryable storage columns or filter via JSON-path on the persisted `metadata.provenance[0]` shape.
- **D.** Reserved system entry type definitions (the 10 from REQ-d00134) SHALL ship with `EntryTypeDefinition.materialize == false`. No materializer SHALL fire on system events on either the append or ingest path. (Future cross-aggregate stream projections — out of scope for this phase — will use a different abstraction than `Materializer`.)
- **E.** `EventStore.ingestBatch` and `EventStore.ingestEvent` SHALL NOT call any method on `DestinationRegistry`, `EntryTypeRegistry`, or FIFO state. Bridged system audit events are stored in `event_log` only; the receiver-stays-passive invariant (§3.5) is permanent.
- **F.** When a destination's `SubscriptionFilter.includeSystemEvents == true`, the destination SHALL receive every event whose `entry_type` is in `kReservedSystemEntryTypeIds` and whose `client_timestamp` is within the destination's active time window (REQ-d00129-I), regardless of `entryTypes` content. When `false`, system entry types are rejected before `entryTypes` is consulted.

REQ parent: `REQ-p00004` (Immutable Audit Trail via Event Sourcing) and `REQ-p01001` (Offline Event Queue with Automatic Synchronization).
