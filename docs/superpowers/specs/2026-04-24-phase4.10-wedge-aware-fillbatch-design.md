# Wedge-Aware fillBatch Skip (Phase 4.10 Design Spec)

**Date**: 2026-04-24
**Ticket**: CUR-1154 (continuation of the `mobile-event-sourcing-refactor` branch)
**Phase**: 4.10 — follows Phase 4.9 (sync-through ingest). Library-only, additive efficiency change.
**Scope**: `event_sourcing_datastore` library — early-return in `fillBatch(destination)` when the destination's FIFO head is wedged. One new assertion on REQ-d00128. No new REQ number. No consumer apps are touched.

## 1. Problem

Today `fillBatch(destination)` runs on every `syncCycle` tick. It walks the event log past `fill_cursor`, filters by the destination's subscription filter and time window, and (when matches exist) promotes them into a fresh FIFO row. There is no awareness of whether the destination's FIFO is currently wedged.

When a FIFO is wedged, drain halts at the wedged head per REQ-d00124-H. Any FIFO row `fillBatch` writes behind that head sits idle until the operator runs `tombstoneAndRefill` (REQ-d00144). On `tombstoneAndRefill`:

- The wedged head becomes `tombstoned`.
- Every pending row strictly after it on the queue is deleted (REQ-d00144-C: `deleteNullRowsAfterSequenceInQueueTxn`).
- `fill_cursor` is rewound to the last `sent` row's `sequence_number` (or `-1` if none), so the next `fillBatch` re-promotes the events covered by the tombstoned target AND by the deleted trail into fresh bundles against the current transform and destination state.

The deleted trail is the speculative work: every event that landed during the wedge was bundled once into a row that was immediately stranded behind the wedge, then deleted on recovery, then re-bundled into a fresh row on the next `fillBatch`. Two `fillBatch` invocations per event, only the second of which produces a row that drain ever ships.

The cost is bounded but not zero. Each speculative bundle costs `Destination.transform(batch)` plus a FIFO row write plus a `fill_cursor` advance, all inside one storage transaction. Wedges can persist for hours or days (the recovery model is "operator notices, ships an app update, then runs `tombstoneAndRefill`"); event arrival continues during the wedge. The longer the wedge, the more speculative rows accumulate to be deleted on recovery.

## 2. Design

### 2.1 Behavior change

`fillBatch(destination)` reads `backend.readFifoHead(destination.id)` as the first I/O of the call. If the returned row's `final_status == FinalStatus.wedged`, `fillBatch` returns without doing anything else: no event-log walk, no transform call, no FIFO row write, no `fill_cursor` advance.

The early-return is structurally identical to the existing dormant-schedule and closed-window early-returns at the top of the function. From the caller's perspective (`sync_cycle`, integration tests, the demo app) the call is a void no-op; `fillBatch` retains its `Future<void>` return type. Observability of the wedge state remains in the existing read surface (`readFifoHead`, `wedgedFifos`, `anyFifoWedged`).

If `readFifoHead` returns `null` (FIFO empty) or returns a row with `final_status == null` (pre-terminal, drain-eligible), `fillBatch` proceeds to its existing behavior — read `fill_cursor`, walk candidates, promote a batch if any match. The check is exactly "head is present AND its `final_status` is `wedged`."

### 2.2 Cursor handling on wedge-skip

The wedge-skip branch does NOT advance `fill_cursor`. The motivation is identical to the rest of the design: zero work on the wedged tick. `readFifoHead` is the only I/O.

`tombstoneAndRefill` rewinds `fill_cursor` itself (REQ-d00144). Whatever value the cursor held at wedge-time is overwritten by the recovery; whether `fillBatch` would have advanced it during the wedge is moot. After the rewind, the next `fillBatch` re-walks from the rewound cursor and promotes everything in one pass.

`tombstoneAndRefill` is the in-spec recovery primitive for a wedged head. The library codebase carries some legacy `wedged → null` "rehabilitate" plumbing in `StorageBackend.setFinalStatusTxn` and friends marked as implementing REQ-d00132, but no `REQ-d00132` section currently exists in `spec/dev-event-sourcing-mobile.md` (the spec jumps from REQ-d00130 to REQ-d00133). Phase 4.10 does NOT depend on rehabilitate semantics, does NOT cite REQ-d00132 in any new artifact, and does NOT extend the wedge-skip optimization to consider that legacy path. Reconciling the broken REQ-d00132 references in lib code with the absent spec section is out of scope for Phase 4.10 and is flagged in `PHASE_4.10-4.13_DECISIONS_LOG.md` for user review.

### 2.3 Per-destination scope

`fillBatch` already takes one `Destination` and operates on that destination's FIFO and cursor in isolation. Multi-destination is `sync_cycle`'s loop, which calls `fillBatch` once per destination per tick. A wedge on destination A causes `fillBatch(A)` to return early; `fillBatch(B)` on the next iteration of the loop is unaffected and runs normally. This falls out of the existing per-destination structure without additional code.

### 2.4 Where the events live during the wedge

Origin events continue to land in `event_log` via `EntryService.record` regardless of FIFO state. The wedge-skip changes only the FIFO-promotion side of the pipeline; the authoritative store (`event_log`) is untouched. After `tombstoneAndRefill` rewinds the cursor, the next `fillBatch` walks `event_log` from the rewound cursor and produces fresh FIFO rows covering the events that landed during the wedge. The wedge-skip eliminates the speculative bundle/delete/rebundle round-trip; end-state after recovery is identical to a system that promoted-then-deleted speculatively, just without the wasted work.

## 3. REQ changes

### REQ-d00128 (FIFO Batch Shape and Fill Cursor)

One new assertion appended after the existing H. No renumbering of A–H; new assertion is I.

> I. `fillBatch(destination)` SHALL return without promoting any events when `backend.readFifoHead(destination.id)` returns a row whose `final_status` is `FinalStatus.wedged`. The early return SHALL NOT advance `fill_cursor` and SHALL NOT call `Destination.transform`. Recovery via `tombstoneAndRefill` (REQ-d00144) rewinds `fill_cursor` so the next `fillBatch` promotes the covered events in one pass against the current transform and destination state.

Rationale paragraph appended to REQ-d00128: a brief note that drain halts at a wedged head (REQ-d00124-H), so any FIFO row promoted behind a wedged head is speculative work that `tombstoneAndRefill`'s trail-delete sweep would have to undo. Skipping promotion during wedge eliminates the speculative bundle/delete/rebundle round-trip without changing end-state.

### No other REQ changes

- REQ-d00124-H (drain halts on wedged head) is unchanged and is the reason for I, not a target of modification.
- REQ-d00144 (tombstoneAndRefill) is unchanged. Cursor handling under tombstoneAndRefill recovery is described in §2.2 of this spec for narrative completeness; the recovery op itself does not move.
- REQ-d00125 (sync_cycle) is unchanged. The per-destination loop already invokes `fillBatch` and `drain` independently per destination.

### REQ-number math

Phase 4.9 claimed REQ-d00145 (ingest contracts) and REQ-d00146 (verification APIs). Phase 4.10 extends REQ-d00128 with assertion I; no new REQ number is claimed. Phase 4.11 will claim REQ-d00147 and REQ-d00148.

## 4. Out of scope

- **Diagnostic logging on wedge-skip.** Tracing every wedged-tick fill-skip would produce N log lines per minute per wedged destination for the duration of the wedge — noisy, and the existing `wedgedFifos` / `anyFifoWedged` reads already let callers surface "destination is wedged" once on transition rather than once per tick. If a future operator dashboard wants per-tick visibility it can poll those reads.
- **A typed `FillBatchOutcome` return value.** Considered and rejected. `fillBatch` already has three "did nothing" early-returns (dormant, closed window, no matches) and they are all `Future<void>`. Adding a typed result for one of them would either leave the existing three on `void` (inconsistent surface) or churn four call sites for one new branch. Wedge-state observability lives in `readFifoHead` / `wedgedFifos` / `anyFifoWedged`.
- **Reconciling the legacy `REQ-d00132` rehabilitate plumbing in lib code with the absent spec section.** Pre-existing in the codebase before Phase 4.10 began. Surfaced for user review in `PHASE_4.10-4.13_DECISIONS_LOG.md`; not in scope for this phase.
- **Historical replay (REQ-d00130).** Replay runs at `setStartDate` time on a fresh-or-already-quiescent destination. A wedge on the same destination during replay would be a programming error (replay completes before the destination is live for drain). No code change to the replay path; no test for the wedge-during-replay case.
- **`fillBatch` behavior change on rows whose `final_status` is `null` past a wedged head.** That state is unreachable after this change — `fillBatch` skips promotion while wedged, and `tombstoneAndRefill`'s sweep deletes any `null` rows already past a wedged head before recovery completes. No spec text needed.
- **UX consequence of empty pending trail behind a wedged head.** Today the demo `FifoPanel` shows operators a growing pending trail behind a wedged head as a visible signal of "events piling up." After this change the trail stays empty. If the demo wants a "potential pending" count, it can compute one from `event_log` + `subscription_filter` + `fill_cursor` without materializing rows. Not blocking for the library change. Phase 4.10's plan should call out the option for the demo, not implement it.

## 5. Test shape

Two new tests in `apps/common-dart/event_sourcing_datastore/test/sync/fill_batch_test.dart`, both verifying REQ-d00128-I:

1. **Wedge-skip writes nothing.** Set up a destination with one wedged FIFO row at the head. Append several matching events to `event_log`. Call `fillBatch`. Assert: no new FIFO rows; `fill_cursor` unchanged; `Destination.transform` not called (use a tracking fake destination).

2. **Post-`tombstoneAndRefill` fill promotes everything in one pass.** Set up a wedged FIFO row covering events 1–3. Append matching events 4 and 5 (which the live system would normally try to bundle behind the wedge). `fillBatch` is a no-op. Run `tombstoneAndRefill` on the wedged head. Call `fillBatch`. Assert: events 4 and 5 land in fresh FIFO rows together with whatever events the rewound cursor re-covers (per `tombstoneAndRefill`'s rewind semantics), and the fill is one or two `fillBatch` invocations rather than per-event.

Existing `fill_batch_test.dart` and integration tests (`end_to_end_test.dart`, `strict_order_regression_test.dart`) MUST continue to pass — none of them set up a wedged head before calling `fillBatch`, so the new early-return does not affect their assertions.

## 6. Implementation footprint

One file mutation: `apps/common-dart/event_sourcing_datastore/lib/src/sync/fill_batch.dart`. The check goes at the top of `fillBatch`, after the dormant and closed-window early-returns and before the `readFillCursor` call:

```dart
final head = await backend.readFifoHead(destination.id);
if (head?.finalStatus == FinalStatus.wedged) return;
```

Plus an `import` for `FinalStatus` if not already present.

## Requirements

This design implements the wedge-aware short-circuit on `fillBatch` and supporting trail-cleanup invariants. The relevant REQ surface (already in the assertion graph; this section is back-fill for §7 of CLAUDE.md):

- REQ-d00128-I — wedge-aware early-return in `fillBatch` (the central new assertion this spec adds).
- REQ-d00128-E+F+G+H — existing `fillBatch` contract this design extends without changing.
- REQ-d00124-H — operator-recovery wedge-halt invariant the new short-circuit honors.
- REQ-d00130-A+B — historical replay invariants (touched only inasmuch as the wedge guard sits before any candidate scan).
- REQ-d00132-A+B — FIFO trail cleanup expectations (related but unchanged).
- REQ-d00144-A — `tombstoneAndRefill` recovery primitive the wedge guard avoids speculating against.
- REQ-d00145-* — ingest contract (out of scope; cited only for context on receiver-side guarantees).

No new REQ entries land in the assertion graph from this spec; the assertion-level changes (REQ-d00128-I) were merged through the elspais MCP in the same branch as the implementation.
