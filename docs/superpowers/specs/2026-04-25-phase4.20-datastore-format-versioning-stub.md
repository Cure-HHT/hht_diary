# Phase 4.20: Datastore Format Versioning (stub)

**Date:** 2026-04-25
**Status:** Stub — full design deferred. Phase 4.16 stamps the `lib_format_version` field on every event; Phase 4.20 designs how to *evolve* that version (and the existing `batch_format_version`) over time.
**Depends on:** Phase 4.16 ships the field; Phase 4.20 ships the migration mechanism.
**Requirements:** deferred until full design — REQs will be claimed during Phase 4.20's brainstorm + spec.

## Intent

The lib has two internal schemas of its own — the on-the-wire batch envelope and the on-disk `StoredEvent` row — both of which will need to evolve over time. Phase 4.20 designs how that evolution happens without breaking deployed nodes (mobile clients shipped under an older lib, portal instances shipped under another lib version, persisted databases written under an older lib).

## The two axes covered here

These are distinct from the entry-type-schema axis (Phase 4.16 + 4.19), which is application-controlled. The two axes Phase 4.20 covers are lib-controlled — a lib release bumps them in lockstep with code changes, and consumers don't get to register their own versions.

### A. Wire format (`batch_format_version`)

Versions the JSON shape of `BatchEnvelope` carried between mobile and portal. Already half-designed:

- Field exists: `BatchEnvelope.batchFormatVersion` (String, currently `'1'`)
- `BatchEnvelope.currentBatchFormatVersion = '1'` constant produces every outgoing envelope
- `BatchEnvelope.decode` validates: anything other than `'1'` throws `IngestDecodeFailure('unsupported batch_format_version: got X; expected "1"')` (`batch_envelope.dart:33-39`)
- REQ-d00145-B governs the canonical version

What's NOT designed:

- Compatibility window — how long does a receiver support old wire versions before refusing them?
- Forward-compatibility — does a v1 receiver getting v2 bytes reject (current behavior), upgrade in place via a registered decoder, or call back to the sender for a re-encode?
- Negotiation — is there a handshake before bulk send, or does the sender always pick the latest and let the receiver reject?
- Mixed-fleet operation — what's the operator playbook when half the mobile fleet is on lib v1 and the portal is on v2 (or vice versa)?
- Backwards-compatibility — does a v2 sender ever produce v1 bytes for a known-v1 receiver, or do we require all-receivers-upgrade-first?

### B. Storage format (`StoredEvent` on-disk shape)

Versions the field set persisted on disk for each event row.

**What Phase 4.16 already lands** (no longer 4.20's job):

- Top-level `lib_format_version: int` field on every `StoredEvent` (per-row marker, lib-stamped from a constant at `append` time).
- Field included in the event's canonical JSON for hash chaining.
- Field propagated over the wire inside `BatchEnvelope.events[]`.
- Receive-side validation in `ingestBatch`: rejects events whose `lib_format_version` exceeds the receiver's `currentLibFormatVersion` (asymmetric, like `entry_type_version`).

**What Phase 4.20 still needs to design:**

- On-read upgrade path — when `currentLibFormatVersion` bumps from N to N+1, does the lib transparently upgrade old rows when they're read (lazy), require a one-time migration sweep at db open (eager), or refuse to open old dbs entirely (lockstep)?
- Hash-chain interaction — if an upgrade modifies the canonical JSON of an event, the original `eventHash` no longer validates the new shape. Options: preserve the original hash and record the upgrade in provenance; recompute under the new shape with a system audit event noting the rewrite; or version the hashing itself.
- Materializer rebuild — old materialized-view rows derived from old-shape events: do they auto-invalidate on lib upgrade? Or does the rebuild path need to know which view rows correspond to which event versions?
- Sembast db-header marker (vs. only per-row markers) — useful for fast db-open compatibility checks; do we want both?

## Why a separate phase from 4.16 / 4.19

- 4.16 + 4.19 version the *content* of `data.answers` per entry type. The lib treats `answers` as an opaque map; the app and widgets define the schema. Promoter callbacks live in app code.
- 4.20 versions the *envelope* the lib itself defines: the wire JSON shape and the storage row shape. Promoter (or upgrader) logic lives in lib code, shipped with the lib version that introduces the new shape.

These are different ownership boundaries. Conflating them would tangle "I, the app, registered a promoter for epistaxis_event v1→v2" with "I, the lib, ship a built-in upgrader for StoredEvent disk shape v3→v4."

## Anchor points (do not relitigate in 4.20's design)

- REQ-d00145-B: single canonical `batch_format_version`.
- REQ-d00115-G+H+I: hash chain semantics. Any storage-format upgrade must square with these.
- Project memory `project_greenfield_status.md`: mobile + portal are both greenfield, no prior version to migrate from. **Phase 4.20 only becomes urgent when the lib is shipped to a deployed audience, OR when the lib team wants to make a breaking change to `StoredEvent`'s field set during development.**
- Project memory `project_event_schema_migration_strategy.md`: chain-of-promoters pattern (this is for app-controlled entry-type promoters; 4.20 may borrow the *pattern* but the registry is lib-internal).

## Out of scope for 4.20

- Per-entry-type schema versioning (that's 4.16).
- Per-entry-type promoter chains (that's 4.19).
- Materializer interface evolution (separate concern; if `Materializer.applyInTxn`'s signature changes, that's a lib breaking change covered by ordinary release notes, not a runtime upgrade mechanism).
- Sembast db file format changes — those are sembast's concern, not ours.

## Likely shape of the eventual design (sketch only)

For wire format: pick a **conservative receiver-rejects** posture by default (current behavior is correct), plus add a coordinated-rollout convention to the operator runbook: "bump lib version on portal first, then mobile fleet, with a brownout window during which the portal accepts v=N and v=N+1 simultaneously." A lib release that bumps `batch_format_version` would land with both encoder and decoder for the new shape in the same commit.

For storage format: the `lib_format_version` field exists from day one (Phase 4.16). On db open, `SembastBackend` reads the highest version present and refuses to open if it exceeds the lib's `currentLibFormatVersion`. On bumping, the lib ships an upgrader that rewrites old rows in a one-time sweep AND a new event-type-of-its-own (`system.lib_format_upgraded`) that records the upgrade in the audit log. Hash chain across the upgrade boundary is preserved by chaining the old `eventHash` into the upgrade event's `previous_event_hash`.

These are sketches to bound the eventual design space, NOT decisions. Phase 4.20's brainstorm validates or replaces them.
