# Phase 4.19: Event Promoter — Schema-Versioned Materialization

**Date:** 2026-04-26
**Scope:** `apps/common-dart/event_sourcing_datastore/` (lib + tests). Replaces the stub at `2026-04-25-phase4.19-schema-migration-stub.md` (the stub's chain-of-single-step-promoters framing is superseded by the simpler opaque-callback model below).
**Depends on:** Phase 4.16 (`entry_type_version` per event) is shipped. Phase 4.17/4.18 are independent.

## Summary

Every materialization (incremental update *and* `rebuildView`) projects each event through a caller-supplied `EntryPromoter` callback before the materializer's fold runs. The promoter takes `(entryType, fromVersion, toVersion, data)` and returns transformed `data`; the lib does not interpret the transformation, does not compose chains, does not enforce direction or step size. The lib stores per-`(viewName, entryType)` target version metadata so incremental updates know what version to project into. `rebuildView` accepts an explicit `Map<String, int>` of target versions, and bootstrap requires the same map upfront so the metadata is initialized before any event arrives. All merges normalize.

## Architecture

```text
EntryPromoter — opaque caller-supplied callback. The lib doesn't compose,
doesn't validate direction, doesn't enforce single-step.

typedef EntryPromoter = Map<String, Object?> Function({
  required String entryType,
  required int fromVersion,
  required int toVersion,
  required Map<String, Object?> data,
});

The lib's contract:  always call promoter before applyInTxn,
                     even when fromVersion == toVersion,
                     even on rebuild,
                     pass result as `promotedData`,
                     don't wrap thrown exceptions.

Promoter's contract: return promoted data, or throw.
                     Free to walk forward, backward, multi-step,
                     or no-op. Free to reshape.
                     (Caller's internal concern.)

view_target_versions — small lib-owned metadata table.
  PK (view_name, entry_type) -> target_version: int

  Read on every incremental materialization to know `toVersion`.
  Written by:
    - bootstrap (initial map, mandatory per materializer)
    - rebuildView (full-rebuild only; no partial rebuilds)
```

**Invariants:**

- `Materializer.targetVersionFor(entryType)` is no longer "the materializer hardcodes what version to project to" — it's "the materializer reads the lib-stored metadata and returns whatever's recorded." The materializer is a stateless contract; the per-(view, entryType) target lives in `view_target_versions`.
- The promoter is **always** invoked, regardless of whether `fromVersion == toVersion`. The user explicitly opted into this — it gives the promoter the freedom to reshape data even on same-version events.
- The promoter is **opaque to the lib**. No registration, no composition, no validation.
- Thrown exceptions from the promoter **propagate as-is** through `applyInTxn`, which (per REQ-d00140-E) rolls back the entire append/rebuild transaction.
- All merges normalize. There's no "raw" path; even tests must supply *some* promoter (an identity helper is provided).
- `rebuildView` is full-rebuild only. The map of target versions must cover every entry type currently in `view_target_versions` for that view (or a strict superset, allowing new entry-types to be introduced as part of a rebuild). Partial rebuild → `ArgumentError`.

## Components

All changes are in the lib (`apps/common-dart/event_sourcing_datastore/lib/src/`), plus mechanical test/demo updates.

### 1. `EntryPromoter` typedef + identity helper

New file: `lib/src/materialization/entry_promoter.dart`.

```dart
/// External callback invoked by the lib before every materialization fold.
///
/// The lib calls this with the event's authoring `entryTypeVersion` as
/// `fromVersion` and the view's stored target version (per
/// `view_target_versions`) as `toVersion`, regardless of whether they're
/// equal. The returned map is passed to the materializer as `promotedData`.
///
/// The lib treats this function as opaque: it does not compose chains,
/// inspect the result, or interpret `fromVersion`/`toVersion` direction.
/// A thrown exception propagates through the materialization pipeline and
/// rolls back the transaction (per REQ-d00140-E).
typedef EntryPromoter = Map<String, Object?> Function({
  required String entryType,
  required int fromVersion,
  required int toVersion,
  required Map<String, Object?> data,
});

/// Identity promoter — returns `data` unchanged. Useful for tests and for
/// materializers whose registered targets always equal authoring versions.
Map<String, Object?> identityPromoter({
  required String entryType,
  required int fromVersion,
  required int toVersion,
  required Map<String, Object?> data,
}) => data;
```

Public exports added to `event_sourcing_datastore.dart`: `EntryPromoter`, `identityPromoter`.

### 2. `Materializer` contract — two new required members + `applyInTxn` parameter

In `lib/src/materialization/materializer.dart`:

```dart
abstract class Materializer {
  String get viewName;

  bool appliesTo(StoredEvent event);

  /// Required. The lib invokes this before every fold.
  EntryPromoter get promoter;

  /// Required if [appliesTo] can return true for any event. The lib calls
  /// the promoter with `toVersion: targetVersionFor(event.entryType)`. The
  /// implementation reads from `view_target_versions` via the backend.
  ///
  /// Throws `StateError` if no target is registered for the (view, entryType)
  /// pair — this signals a missing bootstrap step or partial rebuild and
  /// should fail loud.
  Future<int> targetVersionFor(
    Txn txn,
    StorageBackend backend,
    String entryType,
  );

  Future<void> applyInTxn(
    Txn txn,
    StorageBackend backend, {
    required StoredEvent event,                    // raw event — keep for clientTimestamp etc.
    required Map<String, Object?> promotedData,    // promoter's output (lib always invokes)
    required EntryTypeDefinition def,
    required List<StoredEvent> aggregateHistory,   // raw, NOT promoted
  });
}
```

The `targetVersionFor` default implementation lives on `Materializer` (concrete, not abstract):

```dart
Future<int> targetVersionFor(Txn txn, StorageBackend backend, String entryType) async {
  final stored = await backend.readViewTargetVersionInTxn(txn, viewName, entryType);
  if (stored == null) {
    throw StateError(
      'Materializer "$viewName": no view_target_version registered for '
      'entry type "$entryType". Either bootstrap with a target version map '
      'covering this entry type, or rebuildView with a map that includes it.',
    );
  }
  return stored;
}
```

So concrete materializers don't usually override `targetVersionFor`; they only override `viewName`, `appliesTo`, `promoter`, and `applyInTxn`.

### 3. `DiaryEntriesMaterializer` — accepts a required promoter, fold operates on `promotedData`

In `lib/src/materialization/diary_entries_materializer.dart`:

```dart
class DiaryEntriesMaterializer implements Materializer {
  const DiaryEntriesMaterializer({required this.promoter});

  @override
  final EntryPromoter promoter;

  @override
  String get viewName => 'diary_entries';

  @override
  bool appliesTo(StoredEvent event) => event.aggregateType == 'DiaryEntry';

  @override
  Future<int> targetVersionFor(Txn txn, StorageBackend backend, String entryType) async {
    // Default impl from Materializer; explicit override only if a concrete
    // materializer wants different semantics (none expected).
    final stored = await backend.readViewTargetVersionInTxn(txn, viewName, entryType);
    if (stored == null) {
      throw StateError('...');
    }
    return stored;
  }

  @override
  Future<void> applyInTxn(
    Txn txn,
    StorageBackend backend, {
    required StoredEvent event,
    required Map<String, Object?> promotedData,
    required EntryTypeDefinition def,
    required List<StoredEvent> aggregateHistory,
  }) async {
    final priorRowRaw = await backend.readViewRowInTxn(txn, viewName, event.aggregateId);
    final prior = priorRowRaw == null ? null : DiaryEntry.fromJson(priorRowRaw);
    final firstTs = aggregateHistory.isEmpty
        ? event.clientTimestamp
        : aggregateHistory.first.clientTimestamp;
    final next = foldPure(
      previous: prior,
      event: event,
      promotedData: promotedData,        // <- new
      def: def,
      firstEventTimestamp: firstTs,
    );
    await backend.upsertViewRowInTxn(txn, viewName, next.entryId, next.toJson());
  }

  static DiaryEntry foldPure({
    required DiaryEntry? previous,
    required StoredEvent event,
    required Map<String, Object?> promotedData,    // <- new
    required EntryTypeDefinition def,
    required DateTime firstEventTimestamp,
  }) {
    final eventAnswers = _extractAnswers(promotedData);   // <- reads promotedData['answers']
    // ... rest of fold unchanged: switch on event.eventType, call mergeAnswers, etc.
  }
}

Map<String, Object?> _extractAnswers(Map<String, Object?> promotedData) {
  final raw = promotedData['answers'];
  if (raw is Map) {
    return Map<String, Object?>.unmodifiable(Map<String, Object?>.from(raw));
  }
  return const <String, Object?>{};
}
```

The merge logic (`mergeAnswers`) is unchanged — it still does the key-wise merge with absent-vs-null distinction (REQ-d00121-J). It just operates on `promotedData['answers']` instead of `event.data['answers']`.

### 6. `rebuildView` signature change

In `lib/src/materialization/rebuild.dart`:

```dart
Future<int> rebuildView(
  Materializer materializer,
  StorageBackend backend,
  EntryTypeDefinitionLookup lookup, {
  required Map<String, int> targetVersionByEntryType,
}) async {
  return backend.transaction((txn) async {
    // 1. Strict-superset check against current stored targets.
    final existing = await backend.readAllViewTargetVersionsInTxn(txn, materializer.viewName);
    for (final entryType in existing.keys) {
      if (!targetVersionByEntryType.containsKey(entryType)) {
        throw ArgumentError(
          'rebuildView("${materializer.viewName}"): partial rebuild not '
          'supported. Missing entry type "$entryType" (currently registered '
          'at version ${existing[entryType]}). Either include it in '
          'targetVersionByEntryType, or remove it from the view by '
          'modifying Materializer.appliesTo and calling clearViewTargetVersionsInTxn '
          'separately.',
        );
      }
    }

    // 2. Clear the view rows AND the metadata.
    await backend.clearViewInTxn(txn, materializer.viewName);
    await backend.clearViewTargetVersionsInTxn(txn, materializer.viewName);

    // 3. Write new target versions.
    for (final entry in targetVersionByEntryType.entries) {
      await backend.writeViewTargetVersionInTxn(
        txn,
        materializer.viewName,
        entry.key,
        entry.value,
      );
    }

    // 4. Replay events through the materializer with the new targets.
    final events = await backend.findAllEventsInTxn(txn);
    final aggregateIds = <String>{};
    final historyByAgg = <String, List<StoredEvent>>{};
    for (final event in events) {
      if (!materializer.appliesTo(event)) continue;
      final def = lookup.lookup(event.entryType);
      if (def == null || def.materialize == false) continue;
      if (!targetVersionByEntryType.containsKey(event.entryType)) {
        // Strict: every entry type the materializer applies to MUST be in the map.
        throw ArgumentError(
          'rebuildView("${materializer.viewName}"): event ${event.eventId} has '
          'entry type "${event.entryType}" which is not in '
          'targetVersionByEntryType. The map must cover every entry type '
          'this materializer can match.',
        );
      }
      final history = historyByAgg.putIfAbsent(event.aggregateId, () => <StoredEvent>[]);
      final promoted = materializer.promoter(
        entryType: event.entryType,
        fromVersion: event.entryTypeVersion,
        toVersion: targetVersionByEntryType[event.entryType]!,
        data: event.data,
      );
      await materializer.applyInTxn(
        txn,
        backend,
        event: event,
        promotedData: promoted,
        def: def,
        aggregateHistory: List<StoredEvent>.unmodifiable(history),
      );
      history.add(event);
      aggregateIds.add(event.aggregateId);
    }
    return aggregateIds.length;
  });
}
```

The existing `rebuildMaterializedView` convenience function (which loops over multiple materializers) gains the same `Map<String, Map<String, int>> targetVersionByMaterializerByEntryType` shape — one map per materializer.

### 7. `bootstrapAppendOnlyDatastore` — initial target-version map

The bootstrap function gains a required parameter:

```dart
Future<AppendOnlyDatastore> bootstrapAppendOnlyDatastore({
  required StorageBackend backend,
  required Source source,
  required List<EntryTypeDefinition> entryTypes,
  required List<Destination> destinations,
  required List<Materializer> materializers,
  required Map<String, Map<String, int>> initialViewTargetVersions,
  // ^^ keyed on materializer.viewName, then entryType, then version.
}) async { ... }
```

Behavior:

1. After registering entry types and destinations, the bootstrap inspects every supplied materializer.
2. For each materializer, it consults `initialViewTargetVersions[materializer.viewName]`.
3. **If null or missing**: throw `ArgumentError('bootstrapAppendOnlyDatastore: no initialViewTargetVersions entry for materializer "${m.viewName}"')`. Strict — every materializer must have a target map.
4. **If present**: write each `(viewName, entryType, version)` triple via `writeViewTargetVersionInTxn`, idempotently. (Idempotent so repeat-bootstraps on existing storage don't error if values match.)
5. **If existing storage already has a target for `(viewName, entryType)` that conflicts with the supplied map**: throw `StateError('bootstrap conflict: stored target $stored, supplied $supplied for ($viewName, $entryType)')`. Forces the operator to resolve via `rebuildView` rather than silently losing the old target.

### 8. `clearAndDropView` — completely remove a view

A view that's no longer needed (e.g., the materializer was deleted from the codebase) leaves orphan `view_target_versions` rows. Add a one-shot helper:

```dart
Future<void> clearAndDropView(StorageBackend backend, String viewName) async {
  await backend.transaction((txn) async {
    await backend.clearViewInTxn(txn, viewName);
    await backend.clearViewTargetVersionsInTxn(txn, viewName);
  });
}
```

Operator-level recovery primitive. Not exposed on the public API surface initially — it's a `tools/` script. (If needed publicly later, promote.)

## Data Flow

```text
INCREMENTAL — EventStore.append → materializers
==============================================
caller -> eventStore.append(entryType: 'epistaxis_event', entryTypeVersion: 5, ...)
  |
  v
backend.transaction((txn) {
  appendEvent(txn, storedEvent)      // existing
  for each materializer m:
    if !m.appliesTo(event): continue
    if def.materialize == false: continue
    target = await m.targetVersionFor(txn, backend, event.entryType)
       -> reads view_target_versions(m.viewName, event.entryType)
       -> StateError if not registered
    promoted = m.promoter(             // ALWAYS called, even if from == target
        entryType: event.entryType,
        fromVersion: event.entryTypeVersion,
        toVersion: target,
        data: event.data,
    )
       -> caller's function; can no-op, walk forward/back, multi-step
       -> throws propagate, transaction rolls back
    await m.applyInTxn(txn, ..., event: event, promotedData: promoted, ...)
       -> materializer's fold uses promotedData['answers'] (etc.)
       -> writes view row
})


REBUILD — rebuildView(m, backend, lookup, targetVersionByEntryType: {...})
==========================================================================
caller -> rebuildView(diaryEntriesMaterializer, ..., targetVersionByEntryType: {
    'demo_note': 2,
    'epistaxis_event': 5,
    'red_button_pressed': 1, ...
})
  |
  v
backend.transaction((txn) {
  // strict-superset check
  existing = readAllViewTargetVersionsInTxn(txn, 'diary_entries')
  for k in existing.keys:
    if k not in targetVersionByEntryType:
      throw ArgumentError(missing entry type k)

  // wipe
  clearViewInTxn(txn, 'diary_entries')
  clearViewTargetVersionsInTxn(txn, 'diary_entries')

  // re-record metadata
  for entry in targetVersionByEntryType:
    writeViewTargetVersionInTxn(txn, 'diary_entries', entry.key, entry.value)

  // replay
  events = findAllEventsInTxn(txn)
  for event in events:
    if !m.appliesTo(event): continue
    def = lookup(event.entryType)
    if def == null or def.materialize == false: continue
    if event.entryType not in targetVersionByEntryType:
      throw ArgumentError(unmapped entry type)
    promoted = m.promoter(entryType, event.entryTypeVersion,
                          targetVersionByEntryType[event.entryType], event.data)
    await m.applyInTxn(txn, ..., event, promotedData: promoted, ...)
})
```

## Error Handling

### 1. Promoter throws

Propagates through `applyInTxn`, hits `EventStore.append`'s transaction (or `rebuildView`'s transaction) and rolls back per REQ-d00140-E. The original exception type is preserved — the lib doesn't wrap. Caller sees their own promoter's exception and can attach context.

### 2. No target version registered for `(viewName, entryType)` on incremental update

`Materializer.targetVersionFor` throws `StateError` with a message naming the missing pair. Append fails. This is the symptom of a missed bootstrap step or an entry type added after bootstrap without a corresponding `setViewTargetVersion` call — both regulator-relevant gaps; failing loud is correct.

To support adding new entry types post-bootstrap without a full rebuild, the lib also exposes:

```dart
// On AppendOnlyDatastore:
Future<void> setViewTargetVersion(String viewName, String entryType, int version);
```

Operator/dev calls this once when introducing a new entry type; subsequent appends find the target and proceed.

### 3. `rebuildView` partial-rebuild attempted

`ArgumentError` per Components §6. Caller must pass a map covering every existing `(viewName, entryType)` pair plus any new ones being introduced.

### 4. `bootstrapAppendOnlyDatastore` missing materializer's target map

`ArgumentError`. Strict — every materializer must have an entry. No defaults.

### 5. `bootstrap` with conflicting stored-vs-supplied target

`StateError`. Operator must run `rebuildView` to upgrade, not silent overwrite.

### 6. Materializer with `appliesTo == true` matches an event whose `entryType` isn't in the rebuild map

`ArgumentError` from inside `rebuildView`. Symptoms a stale `appliesTo` predicate or a forgotten entry-type mapping — fails loud, doesn't half-build.

## Requirements

### Existing REQs amended

#### REQ-d00121 — diary_entries Materialization from Event Log

Amend assertions B, C, D to operate on `promotedData` instead of `event.data`:

> **B.** When `event.event_type` equals `"finalized"`, `Materializer.apply` SHALL return a `DiaryEntry` whose `is_complete` is `true` and whose `current_answers` equals the key-wise merge of `previous.current_answers` (or the empty map when `previous` is null) under `promotedData['answers']`: for each key `k` present in `promotedData['answers']`, the merged value SHALL equal `promotedData['answers'][k]` — including when that value is `null` (explicit clear); for each key `k` absent from `promotedData['answers']`, the merged value SHALL equal `previous.current_answers[k]` (prior value preserved). `promotedData` is the lib-supplied result of invoking the materializer's `EntryPromoter` per REQ-d00140-G.
>
> **C.** When `event.event_type` equals `"checkpoint"`, [same merge rule applies to `promotedData['answers']` per assertion B]; `is_complete` is `false`.
>
> **D.** When `event.event_type` equals `"tombstone"`, `Materializer.apply` SHALL return a `DiaryEntry` whose `is_deleted` is `true`; all other fields, including `current_answers` and `is_complete`, SHALL carry over unchanged from the previous row. The promoter is still invoked per REQ-d00140-G even though tombstone events typically carry empty `data`.

Existing assertions A, E, F, G, H, I, J unchanged in semantics. Assertion J's "key absent vs key present with null" rule continues to apply, now to `promotedData['answers']`.

#### REQ-d00140 — Pluggable Materializer Contract

Amend assertion A to add the new required members and the `promotedData` parameter:

> **A.** `Materializer` SHALL be an abstract base class with:
> - `String get viewName`
> - `bool appliesTo(StoredEvent event)`
> - `EntryPromoter get promoter` (required, no default)
> - `Future<int> targetVersionFor(Txn txn, StorageBackend backend, String entryType)` (default impl reads `view_target_versions` and throws `StateError` on missing)
> - `Future<void> applyInTxn(Txn, StorageBackend, {required event, required promotedData, required def, required aggregateHistory})`

Amend assertion D to update `rebuildView`'s signature and add the strict-superset check:

> **D.** `rebuildView(materializer, backend, lookup, {required Map<String, int> targetVersionByEntryType})` SHALL replay the event log into exactly one view; running the function twice on the same log with the same `targetVersionByEntryType` SHALL produce the same view rows (idempotent). The supplied map SHALL be a strict superset of the keys currently stored in `view_target_versions` for this view; missing keys SHALL cause `ArgumentError`. Every event whose `entryType` is matched by `appliesTo` SHALL have a corresponding entry in `targetVersionByEntryType`; an unmapped match SHALL cause `ArgumentError` and roll back the rebuild.

Add new assertions G–L:

> **G.** Before invoking each matching materializer's `applyInTxn`, the lib SHALL invoke that materializer's `promoter` with `(entryType: event.entryType, fromVersion: event.entryTypeVersion, toVersion: <view's current target version per `view_target_versions`>, data: event.data)`. The returned `Map<String, Object?>` SHALL be passed to `applyInTxn` as `promotedData`. The lib SHALL invoke the promoter even when `fromVersion == toVersion`.
>
> **H.** A thrown exception from `EntryPromoter` SHALL propagate through `applyInTxn` without being wrapped by the lib; per REQ-d00140-E, the entire append (or rebuild) transaction SHALL roll back.
>
> **I.** `StorageBackend` SHALL expose `readViewTargetVersionInTxn(txn, viewName, entryType) → int?`, `writeViewTargetVersionInTxn(txn, viewName, entryType, version)`, `readAllViewTargetVersionsInTxn(txn, viewName) → Map<String, int>`, and `clearViewTargetVersionsInTxn(txn, viewName)`. These methods constitute the per-`(viewName, entryType)` target-version metadata that drives the `toVersion` argument supplied to the `EntryPromoter`.
>
> **J.** `bootstrapAppendOnlyDatastore` SHALL accept a required `Map<String, Map<String, int>> initialViewTargetVersions` parameter keyed on `viewName` then `entryType`; the bootstrap SHALL write each triple via `writeViewTargetVersionInTxn` before any event is appended. A missing entry for any supplied materializer SHALL cause `ArgumentError`; a stored-vs-supplied conflict on existing storage SHALL cause `StateError`.
>
> **K.** `AppendOnlyDatastore` SHALL expose `setViewTargetVersion(viewName, entryType, version)` for post-bootstrap registration of a previously-unmapped entry type; the call SHALL be idempotent on no-op writes.
>
> **L.** When a materializer's `targetVersionFor` cannot find a registered version for `(viewName, entryType)`, it SHALL throw `StateError` naming the missing pair; the resulting `EventStore.append` (or rebuild) SHALL fail. No silent fallback to a default version SHALL occur.

## Out of scope

- **Built-in chain composition or single-step registration helpers.** The promoter is opaque — if a caller wants to compose `(v1→v2)`, `(v2→v3)`, `(v3→v4)` step-promoters into one `EntryPromoter`, they write that composer themselves. Lib doesn't ship a registry/chain primitive.
- **Reverse-promoter auto-derivation.** Inverses are non-bijective in general; lib doesn't try to derive them. Callers register reverse logic in their own promoter if they need it.
- **Event-log rewriting.** Phase 4.19 is read-time only; the event log stays untouched. Phase 4.20 covers any future on-disk format upgrade.
- **Lib-shipped `NormalizingMaterializer` for trend / analytics / GDPR-export views.** When a concrete trend view emerges as a use case, design it as its own materializer in a follow-up phase. 4.19's machinery will support it.
- **Per-aggregate target version overrides.** All events of a given `(viewName, entryType)` pair are projected to one version. If a future feature needs per-aggregate or per-event-time target overrides (e.g., "this aggregate stays at v=3 forever"), that's a follow-up requirement.
- **Multi-version registry shape on `EntryTypeDefinition`.** Phase 4.16 has `registeredVersion: int` (single). Phase 4.21 may expand this to `Set<int>` for multi-sponsor concurrency. Phase 4.19's `view_target_versions` is independent of `registeredVersion` and won't change shape with that future expansion.
