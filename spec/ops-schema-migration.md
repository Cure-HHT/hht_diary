# DIARY-OPS-schema-migrate-on-deploy: Schema Advanced On Every Deploy

**Level**: OPS | **Status**: Draft | **Implements**: -
**Refines**: DIARY-PRD-platform-operations-monitoring

## Assertions

A. Every application deploy SHALL apply all pending *Database* migrations before routing traffic to the new revision.

B. A failed migration step SHALL prevent the new revision from receiving traffic, leave the prior revision serving, and raise an operator alert.

C. The migration runner SHALL apply only migrations not yet recorded as applied, in ascending order, and SHALL be a no-op when none are pending.

D. Automatic (build-triggered) and manual deploys SHALL apply migrations identically.

## Rationale

In the 2026-05-10 UAT incident, application code that required schema changes was routed traffic before those changes were applied, causing request failures and partial data corruption. Advancing migrations as a pre-traffic step in every deploy — and blocking promotion on failure — eliminates the race condition that caused the incident. The no-op guarantee (assertion C) ensures the migration step adds negligible overhead on deploys where the schema is already current. Treating automated and manual deploys identically (assertion D) closes the gap that existed in UAT where manual promotion bypassed the migration sequence.

*End* *Schema Advanced On Every Deploy* | **Hash**: 0ad4647e

# DIARY-OPS-db-reset-non-prod: Database Reset Is Manual And Non-Production

**Level**: OPS | **Status**: Draft | **Implements**: -
**Refines**: DIARY-PRD-platform-operations-monitoring

## Assertions

A. Dropping and reapplying the consolidated schema (reset) SHALL occur only by explicit manual trigger, never automatically on deploy.

B. Reset SHALL be refused when the target environment is production.

## Rationale

A *Database* reset destroys all data in the target environment. It is a legitimate tool for development and pre-production environments (e.g. re-seeding a QA environment to a known state), but it has no place in the automated deploy path and must never touch production. The 2026-05-10 UAT incident highlighted the risk of automation operating beyond its intended scope; restricting reset to explicit manual invocation and blocking it entirely on production prevents accidental or automated data destruction in regulated environments.

*End* *Database Reset Is Manual And Non-Production* | **Hash**: bc7f6fc8

# DIARY-OPS-ci-migration-gate: CI Migration Soundness Gate

**Level**: OPS | **Status**: Draft | **Implements**: -
**Refines**: DIARY-PRD-platform-operations-monitoring

## Assertions

A. CI SHALL apply the consolidated baseline to a fresh *Database* and fail if it does not apply cleanly.

B. CI SHALL apply all migrations on top of the fresh baseline and report any migration that conflicts with the baseline.

C. CI SHALL re-apply all migrations and report any migration that is not re-runnable (not idempotent).

## Rationale

The CI migration gate exists to catch schema problems before they reach any deployed environment. Assertion A verifies that the baseline itself is consistent and self-contained. Assertion B ensures that migrations do not duplicate or contradict schema objects already established by the baseline — a class of error that went undetected in the lead-up to the 2026-05-10 UAT incident. Assertion C validates idempotency, which is required for safe re-deployment and rollback scenarios: a migration that cannot be re-run safely is a latent operational hazard. All three checks run against a disposable fresh *Database* in CI so they impose no risk on deployed environments and complete without human involvement.

*End* *CI Migration Soundness Gate* | **Hash**: aa6d3750

# DIARY-OPS-baseline-generated: Consolidated Baseline Is Built, Not Hand-Shipped

**Level**: OPS | **Status**: Draft | **Implements**: -
**Refines**: DIARY-PRD-platform-operations-monitoring

## Assertions

A. The consolidated baseline SHALL be generated by tooling rather than maintained by hand.

B. The migration job SHALL consume the baseline baked into its image rather than a mutable external object.

## Rationale

A hand-maintained baseline drifts silently from the true schema: edits are easy to miss, reviews are difficult to scope, and the result cannot be reproducibly verified. Generating the baseline from tooling (assertion A) makes its content deterministic and auditable — the generation step is the single place where the baseline is defined, and re-running it on the same inputs always produces the same output. Baking the baseline into the image (assertion B) ensures that a given revision's migration job always uses the schema snapshot it was built against, with no dependency on external mutable objects that could change between image build and job execution.

**Implementation status**: The build-time generation and image-baking workflow described by assertion B is implemented. The complementary "freeze source SQL and regenerate" discipline — where the hand-authored source files are replaced by a generated artifact checked into the repository — is a planned follow-up. Reviewers should note that assertion A is partially target-state: the baseline is currently generated at build time from source SQL that is still maintained by hand; the planned step closes the remaining gap.

*End* *Consolidated Baseline Is Built, Not Hand-Shipped* | **Hash**: 364bac4d
